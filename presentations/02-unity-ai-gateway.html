The issue

AI traffic accumulates faster than anyone wires up governance for it.

A typical rollout: a production agent on Databricks Apps, a Genie space, developers using Cursor or another coding agent, a custom MCP server built last quarter, and an external model API a team wired up directly. Each path was built independently, by a different team, at a different time.

AI surfaceWhere its calls are recorded today
Coding agent (per developer)Vendor portal, if the vendor exposes one
Production agent on AppsAgent logs and MLflow traces, if instrumented
Genie spaceIts own dashboard, separate from everything else
Custom MCP serverServer stdout or a bucket somewhere
External model API (direct)Whatever the calling team happened to log

The concrete cost: no single place to see every AI call, guardrails applied on some paths and not others, and no way to answer “what did this cost, and was it safe” without polling every system separately.

What it is

One gateway, three traffic surfaces.

Every AI call — from a developer’s coding tool, a production agent, a tool call to an MCP server, or a request to an external API — routes through the same control point. Same permissions, same audit, same cost surface.

Surface 1

LLMs

Agents, LLM serving endpoints, coding agents (Cursor, Codex CLI, Gemini CLI, Claude Code), and Databricks Apps agents. Native multi-provider APIs — OpenAI, Anthropic, Gemini.

Surface 2

MCPs

External, custom (Databricks Apps), and Databricks-managed MCP servers — registered as Unity Catalog securables, under the same permissions and audit as LLM calls.

Surface 3

Serving endpoints & APIs

Gateway features layered onto existing model serving endpoints, plus arbitrary external HTTP APIs — same rate limits, guardrails, and logging.

One control point. Permissions · rate limits · guardrails · spend caps · usage tracking · inference logging · fallback routing.

What it is · Request flow

A single request, end to end.

Six touchpoints, all on one trace, regardless of which of the three surfaces the call belongs to.

All six touchpoints land in system.ai_gateway.usage and inference tables — one request_id ties the whole trace together.

What it is · Three pillars

Identity, runtime, and observability.

The same three questions apply to every call, on every surface: who is allowed to make it, what happens to it in flight, and what gets recorded.

PillarMechanismWhat it answers
IdentityUnity CatalogWho can access which models, MCP servers, connections, and prompts. EXECUTE grants; same identity model as tables.
RuntimeAI Gateway v2Rate limits, guardrails, service policies (ABAC), spend caps, and routing — applied before the request reaches a model and before the response reaches a caller.
ObservabilitySystem tables + inference tablesWhat happened, who did it, what it cost, and whether it was blocked — joinable to billing, queryable with SQL.

Guardrails are one lever, not the whole story. Service policies extend beyond content filtering to govern tool calls and require human approval — something a content filter alone cannot do.

What it is · Guardrails

Six guardrail templates, evaluated on every call.

An LLM evaluator (recommended: databricks-gpt-5-nano) scores every request and response. Configure per phase (input/output), per action (block/sanitize/alert).

TemplatePhaseActionWhat it catches
PII RedactionInput / OutputSanitizeReplaces PII with placeholders (e.g. [EMAIL]) before the model sees it or the user sees the response.
PII BlockingInput / OutputBlockRefuses the request entirely if the payload contains PII — stricter option for regulated workloads.
Unsafe ContentInput / OutputBlockHate speech, harassment, violence, self-harm, sexual content, weapons, extremism.
JailbreakInput onlyBlockInstruction overrides, obfuscated payloads, role-play exploits, prompt extraction attempts.
HallucinationOutput onlyBlock / AlertOutput unsupported by retrieved context — useful where grounding matters (RAG).
CustomInput / OutputYou chooseYour own evaluator prompt, up to 5,000 characters — domain-specific policy (no internal codenames, no competitor mentions).

Fail-closed by default. A blocked request returns HTTP 400. An evaluator timeout or parse error also blocks — it does not silently pass the request through.

What it is · Where the gateway fits

Four traffic patterns. The gateway is the right answer for exactly one of them.

TrafficPatternGovernance mechanism
Agent ↔ Genie, FM API, Agent Bricks, Vector Search, UC Functions, custom MCP on AppsNo gatewayOBO or M2M, direct — UC row filters and column masks enforce at the data plane
LLM endpoint traffic needing rate limits, guardrails, or cost trackingAI GatewayDatabricks AI Gateway configured on the endpoint or as a standalone gateway endpoint
Agent calling external APIs, third-party MCP servers, or external LLMsUC ConnectionsUC HTTP Connections + Serverless Network Policy — credential never enters application code
External clients (enterprise apps, partners) calling into DatabricksExternal API gatewayCustomer-managed gateway (APIM, Kong, AWS API Gateway) — additive, sits in front

The one pattern to get right: traffic that’s already internal to Databricks — an agent calling Genie, Agent Bricks, or a custom MCP server — should not route through a gateway. More on why in the How section.

Why it matters · Cost

Spend caps that block, not just alert.

Hard spend caps

Unity AI Gateway supports spend budgets with shared and per-user thresholds. Thresholds can be configured either to alert or, when block usage is enabled, to stop further requests until the budget resets or an admin raises the threshold. Enforcement is near real time, so a small amount of overspend may still occur.

# budget config shape
budgets:
  - scope: shared | per_user
    period: monthly
    limit_usd: 500
    action: alert | block
$18k → $500
an unbounded batch run capped by a hard budget, illustrative example

Unified spend query

Join system.billing.usage to system.ai_gateway.usage by usage_context tags — the same UC query model finance already uses. No export pipeline.

Why it matters · Risk & compliance

One audit trail answers a question that used to take days.

An agent that calls an LLM, then an MCP tool, then another LLM is three governance domains without a gateway. With one, it is a single trace.

Before

“Show me every AI call this org made last week” means checking a coding-agent vendor portal, agent logs, a Genie dashboard, a custom MCP server’s stdout, and whatever the team using an external API happened to log. Days, not minutes — and still incomplete.

After

One query against system.ai_gateway.usage plus inference tables. Every principal, every endpoint, every guardrail decision, joinable to billing. Minutes, not days.

Fail-closed guardrails mean a blocked request is a recorded decision, not a silent pass-through — the audit trail includes what was stopped, not just what succeeded.

Why it matters · Velocity

Swap models and providers without touching application code.

Fallback and traffic splitting

Primary provider fails → gateway routes to a secondary served entity automatically. A/B test a new model on a percentage of traffic. Stage a rollout from one provider to another with built-in comparison metrics — no client code changes.

Coding agents and Apps agents inherit the same controls

Cursor, Codex CLI, Gemini CLI, and Claude Code point at OpenAI-compatible base URLs. A Databricks Apps agent changes one base URL. Both inherit permissions, rate limits, guardrails, usage tracking, and cost attribution — with zero additional wiring.

Native provider APIs, not a translation layer. Each provider’s SDK works unchanged against the gateway’s native surface — streaming, system prompts, structured tool use all behave exactly as the SDK expects.

How · Getting it running

From zero to governed, in one sitting.

1
Enable

Account admin enables Unity AI Gateway from the account console. Workspace-wide flip.

2
Create endpoint

Pick a target served entity (FMAPI, external model, custom). Set rate limits and guardrails inline.

3
Grant access

Add users, groups, and service principals to the endpoint’s permission list — same identity model as UC.

4
Point clients

Update agents, coding tools, and external apps to the gateway base URL. Existing SDKs work unchanged.

Native provider APIs
from openai import OpenAI
client = OpenAI(
  base_url="https://<uag>/v1",
  api_key=DATABRICKS_TOKEN,
)
client.responses.create(...)
from anthropic import Anthropic
client = Anthropic(
  base_url="https://<uag>/anthropic",
  api_key=DATABRICKS_TOKEN,
)
client.messages.create(...)

Same pattern for Gemini (/gemini/v1) and coding agents (/cursor/v1, /codex/v1).

  }
}

When to use the layered path

  • Existing serving endpoints in production — flip on gateway features without re-pointing clients
  • Guardrail needs are covered by safety / PII / keyword filters, not the full six-template catalog
  • You want classic inference-table audit on a custom model endpoint
  • Workload isn’t on the coding-agent / native-API path yet — both paths coexist
How · Configuring runtime controls

Rate limits, spend caps, and access policy — three separate levers.

LeverGovernsDetail
Rate limitsCall volumeQPM and TPM, per user/group/SP/endpoint. Precedence: endpoint ceiling > named SP override > user > group > default. Max 20 limits per endpoint.
Spend capsDollar budgetHard block on threshold, per endpoint/user/account. Not an alert-and-continue — genuinely blocks.
Service policies (ABAC)Access + tool callsAllow, deny, or require human approval — extends beyond content filtering to govern which tools a principal may invoke.

Guardrails are a subset of service policies. Service policies can also gate tool invocation and require an approval step before the agent proceeds — something a content guardrail alone cannot do.

How · Beyond per-call rules

A session-aware policy layer on top of the gateway.

Why a fifth lever

Rate limits, spend caps, and service policies each evaluate one call at a time. Omnigent, the coding-agent harness, adds a policy layer that remembers the session — it accumulates state across every tool call and model response, so a decision can depend on what already happened, not just the current request.

ALLOWProceed as requested
DENYBlock the action outright
ASKPause for human approval
TRANSFORMRewrite the action, then proceed

Intent-based authorization

The policy remembers the user’s stated task for the session and scopes which tools the agent may call to what that task actually needs — tightening further as the session accumulates risk, rather than granting the agent’s full permission set up front.

Cost as a session-state policy

A running spend total is the same mechanism applied to dollars: a soft threshold triggers ASK, a hard cap triggers TRANSFORM to a cheaper model. Full depth in the agent cost governance deck.

Things to know

What breaks quietly if you skip the fine print.

IssueDetail
Gateway placed in the OBO pathDon’t put a gateway in the OBO path. It breaks per-user data access — the gateway re-issues its own token, so current_user() resolves to the gateway’s identity instead of the human’s, and row filters evaluate silently wrong. Keep internal Databricks-to-Databricks traffic direct.
Evaluator timeout treated as pass-throughIt isn’t. Guardrails fail closed — a timeout or parse error blocks the request.
Spend cap assumed to be a soft alertIt’s a hard block. Calls return 429 once the budget is hit, with no manual override until the period resets.
External gateway and AI Gateway treated as either/orThey’re additive. External gateway handles boundary crossing (external identity → Databricks token); AI Gateway handles LLM governance at the endpoint; UC handles data access.
Custom MCP server traffic routed through the gateway by defaultOnly do this for the MCP-as-a-governed-tool pattern. Internal agent-to-agent or agent-to-Genie calls should stay direct.
Production checklist

Before this goes live.

Least-privilege endpoint permissions

Specific groups, not workspace-wide. Coding agents accessible only to developer groups.

Rate limits per principal

QPM and TPM caps, tighter for service principals than for human users.

Guardrails on input and output

PII block + unsafe + jailbreak on input. PII redact + unsafe on output.

Fallback configured for critical endpoints

Multi-entity setup with fallback enabled so provider outages route automatically.

Inference tables in a catalog with retention

Audit-grade UC catalog, lifecycle policy for payload data, ACLs scoped to security and engineering.

Internal OBO traffic verified as direct

Confirm no gateway sits between an agent and Genie, Agent Bricks, or a custom MCP server.
