Orchestration & Tool Governance

Model Serving, MCP servers, AI Gateway, and Databricks Apps — four orchestration pillars, one governance model underneath.

Builds on Identity & Auth Foundations and Authorization by Resource Type — this deck is about what happens when agents and tools compose.

Agenda

What this deck covers

1. The four pillars of AI orchestration and where governance is enforced in each

2. Agent Bricks supervisor/sub-agent identity propagation, and the three agent auth methods

3. MCP's three server types and UC Connections as the outbound governance primitive

4. Unity AI Gateway as the single control surface for LLMs, MCPs, and Model Serving

5. Token federation for automation and external callers

6. Configuration steps, common gotchas, and a production checklist

The issue

Composing agents and tools is easy to build. Keeping governance intact across every hop is not.

A supervisor calls sub-agents, sub-agents call tools, tools call external services — each hop is a place where the caller's identity can be silently lost, or where an ungoverned credential can creep in.

OBO mode

Supervisor auto-forwards the user's token to all sub-agents. current_user() = human email from the supervisor down to the deepest tool call — row filters, Genie RLS, and function auth all see the actual human.

Runtime limits & scope requirements

  • Each sub-agent's tools need their own UC grants — grants don't cascade from the supervisor
  • Genie sub-agents need both genie and dashboards.genie scopes, or a 403

What — MCP on Databricks

Three MCP server types, all governed by Unity Catalog.

Managed

Databricks-hosted

Pre-built connectors: Genie, Vector Search, UC Functions, DBSQL. No code, no infra. Auth: automatic, OBO, or PAT.

Custom

Databricks Apps

Your code, deployed as a Databricks App. Full control over tools and auth — OBO, M2M, or mixed. Auth is OAuth only, PAT not used.

External

Via UC Connections

Third-party MCP servers (Jira, GitHub, Slack) accessed through UC HTTP Connections. Credential-free code; the platform injects auth.

Tool selection priority on name collision: Managed (highest trust) → Custom (explicit business logic) → External (third-party). Configure explicit tool routing in the supervisor when ambiguity exists.

What — UC Connections

Application code never calls an external service directly. UC proxies the call.

Two-layer defense in depth

Network (SNP)Workspace-level FQDN allowlist — anything not listed is unreachable
Credential (UC)USE CONNECTION grant controls which SP can authenticate, enforced before any network traffic
-- Grant access
GRANT USE CONNECTION
  ON CONNECTION github_api
  TO `sp-appeals`;

-- Instant revocation, no redeploy
REVOKE USE CONNECTION
  ON CONNECTION github_api
  FROM `sp-billing`;

Four auth methods, only two give per-user identity externally: Bearer and OAuth M2M share one identity with the external service; OAuth U2M Per User and Managed OAuth give each user their own token on both sides.

What — Traffic patterns

Four traffic patterns, four different governance layers. Not every hop needs a gateway.

PatternCaller → targetGovernance layer
1Agent to Databricks services (Genie, FM API, UC)No gateway. Direct OBO — UC governs at the data plane
2Any caller to an LLM endpointDatabricks AI Gateway. Rate limits, guardrails, usage tracking, fallback routing
3Agent to external services (GitHub, Slack, etc.)UC Connections + SNP. Proxy model, REVOKE = instant removal
4External clients to DatabricksExternal API Gateway. Auth translation, per-tenant rate limiting, versioning

Why pattern 1 skips the gateway: re-issuing a token at a gateway makes current_user() resolve to the gateway service, not the human — row filters then fire as the gateway. UC enforcement happens at the data plane; a gateway can observe HTTP traffic but not which rows were filtered.

What — Unity AI Gateway

One gateway governs LLMs, MCPs, and Model Serving endpoints under the same control surface.

Pillar 1

LLMs

Agents, LLM endpoints, and coding agents (Cursor, Codex CLI, Claude Code) route through gateway endpoints with native multi-provider APIs.

Pillar 2

MCPs

External, custom, and Databricks-managed MCP servers governed under the same permissions, audit, and access controls.

Pillar 3

Model Serving Endpoints

Gateway features layered onto existing endpoints — safety, PII, and keyword guardrails, plus the established system-table observability surface.

One control point: permissions · rate limits · guardrails · usage tracking · inference logging · cost attribution · fallback routing · traffic splitting — applied consistently across every pillar. Coding-agent calls and deployed-agent calls land in the same system.ai_gateway.usage table.

Why it matters

Orchestration governance is where cost, risk, and compliance actually get decided.

Cost control

Rate limits per user, group, or SP cap a runaway agent before it drives an unexpected model bill. system.ai_gateway.usage joined with system.billing.usage attributes spend by team, app, or project.

Risk reduction

REVOKE USE CONNECTION is a runtime kill switch — no redeploy needed to cut off a compromised or misbehaving tool. Guardrails catch PII and unsafe content before it reaches a model or a caller.

Compliance & velocity

One audit trail across coding agents, deployed agents, and tool calls means one place to answer “who called what, and what did it return” — instead of stitching together per-server logs.

Before this unification, MCP governance was ad hoc: UC handled data access, but request-level audit and rate limiting were per-server. Now LLM calls and tool invocations share one permissions model and one configuration surface.

How — Token federation

Configuring federation for automation and external callers: four steps, no secrets stored.

The exchange flow

1App authenticates with your IdP, receives a JWT
2POST the JWT to /oidc/v1/token with grant_type=urn:ietf:params:oauth:grant-type:token-exchange
3Databricks validates the JWT against the federation policy, returns a scoped OAuth token
4Use the Databricks token to call APIs: Authorization: Bearer <token>

Two federation types

AWS IAM and GitHub Actions OIDC: WIF only. Token TTL copies the IdP's exp claim verbatim — plan refresh in client code.

How — Scopes & gotchas

Least-privilege scopes, plus the failures that don't throw a clear error.

SymptomRoot causeFix
Genie returns wrong rows, no erroris_member() in a row filterReplace with current_user() — is_member() evaluates the execution service under Genie/Agent Bricks OBO
User identity lost in App A → App BTwo-proxy strippingUse X-Forwarded-Email (proxy-signed, survives hops) + M2M SQL with an email-based filter
Sub-agent 403 on a Genie toolScope not granted on the sub-agent's own SPGrant UC permissions and OAuth scopes per sub-agent — nothing cascades from the supervisor
External tool stops working post-deployConnection ownership tied to the old SPSet connection ownership to the app SP; re-grant USE CONNECTION after every redeploy

Scope ceiling: a token without serving, genie + dashboards.genie, or mcp.* cannot call that resource, even if UC grants would otherwise allow it. Request only what the agent needs.
