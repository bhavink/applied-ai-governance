Embedding a Dashboard
Behind a Foreign IdP

Embedding an AI/BI Dashboard inside an app whose IdP isn’t the Databricks account’s IdP — without showing the viewer a second login screen.

The problem

A foreign IdP surfaces a second login screen.

When a host application’s IdP differs from the Databricks account’s own SSO IdP, Basic Embedding still sends the viewer’s browser to authenticate directly against the account’s IdP. If the host app already signed that viewer in through a different IdP, the result is a second, unfamiliar login prompt inside the embedded iframe.

Why this matters beyond the extra click

The second prompt is unbranded and unfamiliar to the viewer, and it appears in front of every downstream client using the host app — not just internal users.

The setup

Acme Managed Services Portal

Custom React app. Every user — internal ops and downstream clients — authenticates once, via Azure AD.

The Databricks account behind it

Its account-level SSO / Unified Login is configured against Google Workspace — a different IdP than the Portal’s.

The embed

The dashboard is embedded via a plain iframe. The viewer’s browser, inside that iframe, is asked to authenticate directly — against Google, not Azure AD.

This is expected behavior of Basic Embedding, not a Databricks defect. Basic Embedding is designed to authenticate the viewer directly against the account’s own IdP; when that differs from the host app’s IdP, a second prompt is the designed-for outcome. The fix is a different embedding model, not reconciling the two IdPs.

What — the two embedding models

Databricks offers two embedding models. They handle a foreign IdP differently.

Basic Embedding Embedding for External Users
Who authenticates to Databricks The viewer, directly, against the account’s SSO IdP The app’s own server, using its own Databricks identity
Viewer needs a Databricks identity? Yes — this is the second login prompt No
Where the prompt appears Inside the iframe, whenever the IdP session is stale Never — token is injected server-side
Data access model Viewer’s own UC permissions SP’s UC permissions, narrowed per-query by an app-supplied parameter
Setup Share dialog → generate embed code Publish the dashboard, create an app identity in Databricks, add server code that requests tokens

If the host app’s IdP differs from the Databricks account’s SSO IdP, Basic Embedding cannot avoid a second login — that is how the model is designed to work. The fix is switching to a model where the viewer’s Databricks identity is never in play.

What — the fix, in plain terms

Three actors. The viewer never talks to Databricks.

An app like the Portal has two parts: the page the viewer looks at in their browser, and a program running on a server the company controls, which the viewer never sees directly. That server-side program is what people mean by “the backend.” It is the only one of the three actors below that talks to Databricks.

Viewer’s browser — what the person sees
App’s server (the backend) — code the company runs, invisible to the viewer
Databricks — where the dashboard and data live
Viewer’s browser → App’s server Already signed in once, via Azure AD. Opens a dashboard tab inside the app.
App’s server → Databricks Logs in as itself, using a secret only it knows. (Databricks calls this app identity a service principal, or SP.) Databricks hands back a generic access pass.
App’s server → Databricks Says which viewer this is for, and what slice of data they’re allowed to see. Databricks hands back a ticket scoped to that one viewer.
App’s server → Databricks Trades that ticket for a dashboard-specific key. Databricks hands back the key, good for about an hour.
App’s server → Viewer’s browser Hands the key to the browser.
Viewer’s browser → Databricks Uses the key to load the dashboard directly. No login prompt, from Databricks or anyone else.

The viewer’s browser only ever talks to two things: the app, and Databricks’ dashboard viewer using the key it was handed. It never sees a sign-in screen for either IdP.

Why this matters

Four concrete reasons, not a preference.

No second login for viewers

The viewer’s browser never authenticates to Databricks directly. It only ever talks to the host app and, once handed a scoped key, the dashboard viewer itself.

No need to provision every downstream client

The Service Principal is the only identity Databricks ever sees. Downstream clients can be added to the host app without SCIM-provisioning a single one of their users in Databricks.

Works regardless of the host app’s IdP

Nothing about the token exchange depends on which IdP authenticated the viewer to the host app. Azure AD, Okta, a homegrown login system — the exchange looks identical from Databricks’ side.

Unblocks the AIM / identity-federation ceiling

Automatic Identity Management (AIM), Databricks’ JIT provisioning layer for Entra ID, requires every user, group, and service principal it manages to belong to one Microsoft Entra tenant. Migrating the account’s own SSO IdP to Azure AD only helps a population inside that one tenant. Downstream clients each sit in their own tenant, or none at all, and account-level federation tops out at five trusted issuers — not hundreds. This pattern works independent of any of that.

Why — when to use which

Three patterns, and when to use each.

Use this deck — External-User Embedding

Embedding a dashboard, viewer has no Databricks presence

Host app owns identity end-to-end. Scoping is one attribute (department, client ID, region). No SCIM sync required for viewers.

Use the Cross-IdP Federation deck

Calling Genie, SQL, or Agent Bricks — not a dashboard

Viewers are SCIM-synced and need real current_user() row filters, individually enforced by Unity Catalog, not by application code.

Use this deck — even with a foreign IdP

Branding forbids a Databricks-domain redirect

The viewer never leaves the host app’s domain, and never sees a Databricks or third-party IdP prompt of any kind.

Use Basic Embedding

Internal viewers already on the account’s own SSO IdP

If the host app and the Databricks account already share one IdP, there is no second identity to remove — Basic Embedding is simpler and gives viewers their own real permissions.

How — implementation

The exact calls behind the app server’s three steps.

Verified against Databricks’ reference implementation. This is server code — the viewer’s browser never runs any of it.

Step 1 — SP token
POST {workspace}/oidc/v1/token
Authorization: Basic base64(client_id:secret)

grant_type=client_credentials&scope=all-apis
Step 2 — viewer context
GET {workspace}/api/2.0/lakeview/
  dashboards/{id}/published/tokeninfo
  ?external_viewer_id={email}
  &external_value={scoping_attr}
Authorization: Bearer {step_1_token}
Step 3 — scoped embed token
POST {workspace}/oidc/v1/token
Authorization: Basic base64(client_id:secret)

grant_type=client_credentials
&authorization_details={step_2_response}

authorization_details is Step 2’s response, passed through unmodified. Response: {access_token, token_type, expires_in, created_at}. access_token goes to the embedding SDK — never to the browser as anything else.

external_viewer_id is audit-only. It does not scope access by itself — only external_value, matched in dashboard SQL, does.

How — scoping model

This is application-enforced scoping, not Unity Catalog row filters.

Dashboard SQL — the scoping pattern
SELECT customer_id, region, total_revenue
FROM sales.customer_data
WHERE department = __aibi_external_value

If a query forgets the __aibi_external_value filter, that viewer sees everything the SP can see. There is no platform-level backstop — every query has to be reviewed.

Publish-time decision: individual data permissions (SP holds the union of access, filtering does the narrowing) vs. shared data permissions (the publisher’s access applies to every viewer uniformly — wrong choice whenever viewers must see different data).

Contrast: the Cross-IdP Federation deck’s pattern

Cross-IdP token exchange (Cross-IdP Federation deck)

  • Viewer does get a real Databricks identity (SCIM-synced)
  • current_user() = the human
  • UC row filters fire automatically, for every query, platform-enforced
  • Requires the viewer’s IdP to publish a JWKS (RS256/ES256)

Dashboard external embedding (this deck)

  • Viewer never gets a Databricks identity — SP only
  • current_user() would return the SP, not the human
  • Scoping fires only if the dashboard’s SQL includes the parameter
  • Works with any host IdP — no JWKS requirement at all
How — before you build

Six prerequisites to confirm before implementation.

1
Publish the dashboard

The embedding APIs only work against a published dashboard, not a draft.

2
Choose individual data permissions at publish time

Not shared. This is the setting that makes per-viewer scoping possible at all — and it’s a republish to change later, not a toggle.

3
Identify the scoping attribute up front

What column distinguishes one viewer’s or one client’s data from another’s — and where does that value live today: an IdP claim, the Portal’s own user table, or a Databricks-side mapping table?

4
Create the Service Principal and grant the union of access

Can View on the dashboard, Can Use on the warehouse, UC SELECT across everything any viewer might need — the ceiling that external_value then narrows.

5
Approve the Portal’s own domain for embedding

Workspace Settings → Security → External Access → Embed Dashboards. Add the Portal’s domain — not a Databricks domain. Unrelated to, and fully compatible with, any “no Databricks-branded URL” branding requirement.

6
Confirm third-party cookies are viable for your viewer base

The embedding SDK depends on them. If Acme’s clients run locked-down browser policies, test this before committing to the pattern.
