Federation Pattern · Real-life Walkthrough

How a SaaS embeds AI dashboards for customers without giving them Databricks accounts

External users authenticate with their own identity provider, your app exchanges that identity for a scoped Databricks token, and Unity Catalog enforces per-customer data filtering on every query. No Databricks accounts for end users. No custom row-level access layer in app code.

Scenario

Acme Logistics runs shipping analytics for thousands of retail clients. Each client logs into portal.acme.com with their own Okta or Entra ID and expects to see AI-powered dashboards — refund rates, ETA forecasts, cost-per-shipment — scoped to their own data only.

1 The constraints

No Databricks accountsProvisioning UC users for 10,000+ end customers is unmanageable.
No custom ACL layerFiltering in app code is fragile — one bug exposes another customer's data.
Audit by external identityCompliance must trace activity back to the actual end user, not a shared service principal.

2 Two patterns solve this — same architecture, different productization

PatternUse caseAuth mechanismWhat carries the user identity
RFC 8693 OAuth federation (animation below) Custom apps, agents, MCP servers, anything beyond dashboards External IdP JWT exchanged for a scoped Databricks SP token at /oidc/v1/token SP bound to a UC group → row filters, column masks, USE CONNECTION fire on every query
Databricks AI/BI External dashboard embedding Read-only AI/BI dashboards inside customer portals OAuth M2M (service principal client credentials) + scoped per-viewer token mint external_viewer_id + external_value in the scoped token, used as __aibi_external_value SQL filter

3 How they relate

Both patterns share the same core architecture — only the API surface and use case differ:

Identity inAn external user authenticates against an identity provider your app trusts. The end user never gets a Databricks account.
Scoped token outYour server exchanges that identity for a short-lived, narrowly-scoped Databricks token — either an SP token (RFC 8693) or a per-viewer token (AI/BI embedding).
Server-side enforcementUnity Catalog (RFC 8693) or scoped SQL variables (AI/BI) enforce per-user data filtering at query time — not in app code.

The animation below shows the general OAuth federation pattern — the canonical RFC 8693 token exchange used by custom apps, agents, and MCP servers. Databricks external dashboard embedding is a productized, dashboard-focused variant of the same idea: simpler API surface (M2M + token scoping endpoint instead of full RFC 8693), narrower use case (read-only AI/BI dashboards), but the architectural principle — external identity in, scoped token out, server-side filtering — is identical.
