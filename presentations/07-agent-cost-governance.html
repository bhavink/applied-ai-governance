AI Agent Cost Control
& Governance

How Acme Engineering put six governance layers between their agents and an uncapped cloud bill.

80 engineers · 3 agent types · one control plane · zero billing surprises

The problem

The bill arrives. Nobody expected it.

Last Tuesday, one engineer left a Claude Code agent running overnight to refactor a legacy module. It looped. Three hours. $400. Nobody knew until finance flagged the invoice.
$400
one agent, one night
$0

The quarterly surprise

Finance gets a lump-sum AI line item. Engineering has no idea which team, which agent, which sprint caused it. Chargebacks are impossible. Forecasts are fiction.

Where the spend hides

Claude Code — senior engineers

Autonomous, powerful, expensive. Runs multi-step agentic loops. Can stay busy for hours without a human checkpoint.

Codex-based tools — mid-level engineers

IDE completions at scale. 80 engineers hitting completions constantly adds up faster than anyone models.

Gemini — data team

Notebook generation, data profiling. Token-heavy prompts. Long context windows. One bad notebook template = 200k tokens per run.

All three route through Unity AI Gateway — but without governance configured, it is just a pass-through. The chokepoint exists. The controls do not.

Governance surface

Agents are not just LLM calls.

Every hop in an agent loop is a separate cost event and a separate risk surface. The governance problem is wider than token pricing.

LLM calls

The obvious part

  • Input tokens: context, system prompt, tool schemas, conversation history
  • Output tokens: reasoning, code, tool call arguments
  • Long context loops multiply cost non-linearly
  • Model tier selection per task is uncontrolled
Tool & API calls

The hidden part

  • Agents call search APIs, GitHub, Jira, internal services
  • Each tool result gets fed back into context — more input tokens
  • Unbounded tool loops: agent retries on error, amplifying spend
  • No visibility into which tools are called, how often, by whom
Execution actions

The risky part

  • Shell commands, file writes, git push, terraform apply
  • Agents act on infrastructure — not just generate text
  • No approval gates means an agent can deploy to prod autonomously
  • Audit trail: who approved this? Nothing. The agent decided.
Design principle: Govern the entire agent surface — not just the LLM endpoint. Cost, capability, and safety controls belong at the same layer.

Layer 1 of 6

Model routing & allowlisting.

Not every agent should reach every model. The cheapest effective model for the task saves 60–80% before any other control fires.

Acme policy
Claude Code — Senior

Allowed: Claude Sonnet (default), Claude Opus (explicitly requested)

Blocked: All other model families

Codex Tools — Mid-level

Allowed: Haiku-equivalent tier only

Blocked: Sonnet, Opus, Gemini Pro

Gemini — Data Team

Allowed: Approved Gemini family (flash by default)

Blocked: All Claude families, GPT-4

How it works in the gateway
# Serving endpoint config (simplified)
route_config:
  routes:
    - name: claude-code-senior
      model_allowlist:
        - claude-sonnet-4-*
        - claude-opus-4-*
      tags_filter: {team: engineering, tier: senior}

    - name: codex-mid-level
      model_allowlist:
        - claude-haiku-3-*
      tags_filter: {team: engineering, tier: mid}

    - name: gemini-data
      model_allowlist:
        - gemini-2-flash-*
      tags_filter: {team: data}

Why this matters

An engineer who grabs Opus for a simple docstring task costs 15× more than Haiku for the same output. Without allowlisting, model choice is always whoever pays least attention. The gateway removes the choice.

What the hierarchy enforces

First-match-wins enforcement

The gateway evaluates limits from most-specific to least-specific. An engineer hitting the 60 RPM burst limit gets a 429 immediately — the team or account limit never even needs to fire.

Loop detection in practice

That overnight refactor loop that cost $400? With a 60 RPM per-engineer Claude Code limit, the agent would have hit the burst limit within the first minute of the loop and stopped automatically. Finance sees nothing. Engineer sees a 429 response, investigates, fixes the loop.

Layer 3 of 6

Hard spend caps — the kill switch.

When a cap is hit, calls block. Not degrade. Not slow. Block. The agent stops. The bill does not continue.

Acme spend caps
Scope Cap Period On breach
Account $15,000 monthly 429 — all calls block
Team $2,000 monthly 429 — team blocks
Engineer $50 daily 429 — user blocks

Caps compound with rate limits

Rate limits stop loops in real time. Caps stop cumulative overruns across the billing period. An engineer who spends $50 of Opus at 6am is done for the day — no exception, no override path.

Finance integration

The spend data in system.ai_gateway.usage feeds a nightly roll-up. Finance sees a line-item per team per day. No more quarterly surprises. Chargebacks are automatic.

What Acme achieved

Zero billing surprises in Q3 2026 after caps were configured. The $15k monthly account cap has never been reached — team caps catch overruns first. Engineers know their daily budget. Agents stop themselves.

Layer 4 of 6

Guardrails — what agents can say and do.

Guardrails run on every request and response, before the model sees the input and before the agent sees the output. First match wins.

Input guardrails

Block before LLM

  • Prompts attempting to extract training data
  • System prompt override attempts (Ignore all previous instructions)
  • Direct requests for internal system paths or service credentials
  • Prompt injection via tool results
Fires before token cost is incurred.
Output guardrails

Redact before agent

  • AWS access key patterns (AKIA*)
  • Databricks personal access tokens (dapi*)
  • Private IP ranges in generated code
  • Internal hostnames matching *.acme-internal.io
  • Hardcoded connection strings
Pattern matched before agent consumes output.
PII detection

Redact before model

PII found in agent context (user emails, employee IDs, social security numbers, credit card numbers) is redacted before the LLM processes it.

Redaction is deterministic: the same entity gets the same placeholder across a session so the agent can still reason about it.

john.doe@acme.com → [EMAIL_1]
Required: pii_detection: true on all routes.
Acme configuration: All three guardrail types enabled on every agent route. Guardrails are account-level defaults — no team can opt out without a new policy approval.
Layer 5 of 6

Service policies — approval gates for high-risk actions.

Some tool calls are too consequential to let an agent decide autonomously. Service policies define what requires a human signal before proceeding.

Acme approval matrix
Agent actionAutonomous?Requires
read_file✓ Yes—
run_tests✓ Yes—
create_pr✓ Yes—
git push main✗ NoHuman approval
terraform apply✗ NoHuman approval
delete_resource✗ NoHuman approval

Enforced at the gateway, not in agent code

Service policies live in the gateway configuration, not in each agent's prompt or tool definition. An agent cannot bypass them by being clever. The gateway intercepts the tool call and issues a hold before forwarding.

Policy configuration shape
# Gateway service policy (simplified)
service_policies:
  - name: high-risk-tool-gate
    match:
      tool_names:
        - git_push_main
        - terraform_apply
        - delete_resource
    action: require_approval
    approval:
      timeout_seconds: 300
      approver_groups: [team-leads]
      on_timeout: deny

What the agent experiences

When the gateway intercepts a tool call requiring approval, the agent receives a hold response. Execution pauses. A Slack notification goes to the on-call team lead. Approve → agent continues. Deny or timeout → agent receives a rejection. The entire flow is logged.

Layer 6 of 6

Sandbox isolation — agents that can't escape.

For coding agents specifically, OS-level isolation means the agent's working directory is its entire world. No filesystem escape. No network calls outside the sandbox.

Omnigent — what it adds

Databricks Sandboxes per session

Each Claude Code session gets its own isolated Databricks Sandbox. Filesystem and network are scoped to that session. Credentials are injected at session start, not embedded in the agent prompt.

Session shareable by URL

Any Acme team lead can review a live agent session by opening the session URL. They see exactly what the agent sees: the working directory, the tool call history, the LLM conversation. Full transparency for high-risk sessions.

Spend cap inside the sandbox

The Omnigent policy YAML enforces a per-session token budget. Even if the gateway-level per-engineer cap has headroom, a single session cannot exceed its own sandbox cap.

Policy YAML shape
# Omnigent session policy
sandbox_policy:
  isolation:
    filesystem: workspace_only
    network: allowlist
    network_allowlist:
      - api.github.com
      - jira.acme.io
  credentials:
    injection_mode: env_scoped
    rotation: per_session
  spend:
    token_budget_per_session: 200000
    on_budget_exceeded: terminate
  review:
    shareable_session_url: true
    retain_logs_days: 90

Acme: what changed for engineers

Transparent to normal use. Agents run as before. The difference: when an agent tries to write outside its workspace or call an unlisted API, the sandbox rejects it before the gateway is even involved. Defense in depth.

Layer 6 of 6 — continued

Contextual policies — governance that remembers the session.

A static allow/deny rule judges one action in isolation. A contextual policy judges an action against everything the agent has already done this session — then updates its own state for the next decision.

The state machine

Handler signature

Every contextual policy is a function: it receives the current session state plus the new event, and returns a verdict plus the updated state. The updated state persists for the rest of that session.

Four verdicts, not three

ALLOWProceed as requested
DENYBlock the action outright
ASKPause for human approval
TRANSFORMRewrite the action, then proceed
The same email that would have gone out early in a session may require human approval later — the verdict depends on accumulated session state, not just the action itself.
Built-in contextual policies
PolicyTracksBehavior
Google DriveDocs accessed / createdReading a confidential doc tightens write access afterward (no write-down)
Risk scoreCumulative session riskCrossing a threshold requires approval for sensitive actions
CostCumulative session spendSoft threshold asks; hard cap blocks expensive models
Intent-based authzThe user's initial promptOnly tools relevant to the stated goal are permitted — blocks unrelated or injected calls

Why this matters for Acme

The per-session spend cap in the sandbox policy (previous slide) is itself one contextual policy — a cost tracker with a hard cap. The same mechanism generalizes to risk scoring and intent scoping, without any change to the gateway layer.

Observability

The usage table that answers every question finance has.

system.ai_gateway.usage — every call, every model, every cost, every identity. Queryable like any Unity Catalog table.

Queries Acme runs
-- Daily spend by engineer
SELECT user_id, date_trunc('day', timestamp) AS day,
       sum(total_cost_usd) AS spend_usd,
       sum(total_tokens) AS tokens
FROM system.ai_gateway.usage
WHERE workspace_id = current_workspace_id()
  AND timestamp >= current_date - INTERVAL 30 DAYS
GROUP BY 1, 2
ORDER BY spend_usd DESC;
-- Top 10 most expensive agent sessions
SELECT session_id, user_id, model_id,
       sum(total_cost_usd) AS session_cost,
       sum(total_tokens) AS tokens,
       max(timestamp) - min(timestamp) AS duration
FROM system.ai_gateway.usage
GROUP BY 1, 2, 3
ORDER BY session_cost DESC
LIMIT 10;
-- Model usage by team
SELECT team, model_id,
       count(*) AS calls,
       sum(total_cost_usd) AS spend
FROM system.ai_gateway.usage u
JOIN acme.hr.team_membership t USING (user_id)
GROUP BY 1, 2
ORDER BY spend DESC;
-- Anomaly alert: 3x daily average breach
WITH daily_avg AS (
  SELECT user_id,
         avg(daily_spend) AS avg_spend
  FROM (
    SELECT user_id, DATE(timestamp) AS d,
           sum(total_cost_usd) AS daily_spend
    FROM system.ai_gateway.usage
    WHERE timestamp >= current_date - 30
    GROUP BY 1, 2
  ) GROUP BY 1
)
SELECT u.user_id, today.spend, d.avg_spend,
       today.spend / d.avg_spend AS ratio
FROM daily_avg d
JOIN (...today_spend...) today USING (user_id)
WHERE today.spend > d.avg_spend * 3;

Alert wire-up

The anomaly query runs as a Databricks workflow every 4 hours. Anomalies page the on-call engineer and CC the team lead. No dashboard required — the alert IS the dashboard for urgent cases.

Summary

Six layers, independently configurable.

Defense in depth. Some controls are account-wide. Some are per-team. Some are per-agent type. Each layer can change without touching the others.

Recommended sequence

Don't boil the ocean. Start here.

Each step is independently valuable. Stop after any step if the org isn't ready. The controls compound — later steps become easier once earlier ones are in place.

1
Usage monitoring first — see what's happening

Enable system.ai_gateway.usage and run the spend-by-engineer query. You will be surprised. This one step changes the conversation with finance and engineering leadership. No controls yet — just light.

2
Model allowlists — cheap, high-impact, immediate

Map teams to model tiers based on actual task profiles. Restrict Opus to senior engineers explicitly requesting it. Expect 20–40% cost reduction within the first billing cycle.

3
Rate limits — stop loops, protect burst capacity

Set per-engineer RPM limits conservatively, then tune up. Runaway agent loops disappear. Engineers barely notice the limits during normal use.

4
Spend caps — the kill switch finance needs

Set account, team, and daily-user caps. Finance gets a committed ceiling. Engineering gets autonomy within that ceiling. Both win.

5
Guardrails — protect what leaves the org

Enable PII detection and output secret scanning. Security team owns this layer independently. No engineering changes needed.

6
Approval gates & sandbox isolation — for high-trust agents

Service policies and Omnigent sandboxes for agents with infrastructure access. Deliberate rollout: pilot with one team, tune, then expand.
