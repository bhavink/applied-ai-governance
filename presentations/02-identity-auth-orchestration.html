Foundation

The Three Token Paths

Every resource auth question reduces to one of these. Pick wrong and you lose user identity or introduce shared-credential risk.
OBO — On-Behalf-Of
User's own token flows through the application into the Databricks resource. current_user() returns the human email at every hop — supervisor, sub-agent, SQL engine, row filter.
How: App proxy injects X-Forwarded-Access-Token. Pass as Authorization: Bearer. Inside Model Serving, use ModelServingUserCredentials().
Scopes needed: sql, genie, dashboards.genie, model-serving depending on resource.
Use when per-user data access, row filters, audit trails that name the human.
M2M — Machine-to-Machine
Service principal (SP) authenticates using OAuth client credentials. current_user() returns the SP's UUID — not a human. WorkspaceClient() no-args picks up auto-injected credentials inside Databricks Apps.
How: App resource binding auto-injects DATABRICKS_CLIENT_ID / DATABRICKS_CLIENT_SECRET. SDK picks up automatically.
Gotcha: Row filters using is_member() against the SP see SP's groups, not the user's groups.
Use when background jobs, Vector Search retrieval, shared knowledge bases, Lakebase.
Federation — Token Exchange
External IdP JWT exchanged for a Databricks OAuth token via POST /oidc/v1/token (grant_type=urn:ietf:params:oauth:grant-type:token-exchange). No Databricks secrets needed in external infra.
Account Federation: Users + SPs from your IdP. SCIM required. Up to 5 issuers.
WIF (Workload Identity): CI/CD workloads → SP. No SCIM, unlimited issuers, secretless. M2M only.
Use when cross-IdP multi-agent, CI/CD pipelines, external apps that must not store Databricks secrets.

Serving Endpoints

Serving Endpoints

Agent Bricks, Model Serving, custom inference endpoints

Identity Model

  • OBO: pass X-Forwarded-Access-Token as Authorization: Bearer
  • Inside Model Serving: ModelServingUserCredentials() extracts user identity
  • Scope: model-serving
  • M2M: WorkspaceClient() no-args in Databricks App context
  • current_user() = the token's identity at every resource

Two-Proxy Gotcha

App A → App B: App B's proxy strips App A's forwarded user token, replaces with App A's SP identity. Use X-Forwarded-Email (proxy-signed, cannot be forged) + M2M SQL as workaround.

Agent Bricks OBO Propagation

Supervisor auto-forwards the user's OBO token to all sub-agents. current_user() equals the human email at the supervisor, at each sub-agent, and at each tool resource — up to 20 sub-agents in the chain.

Genie

Genie Spaces

Natural language to SQL, governed by UC. OBO is the only supported path for user-level data enforcement.

Identity Model

  • OBO only — user's token forwarded to Genie API via X-Forwarded-Access-Token
  • Both scopes required: genie AND dashboards.genie
  • current_user() = the calling human — row filters fire correctly
  • is_member() = the execution service, not the human — RLS silently fails
  • Rate limit: 5 queries/min (sliding window)

Row Filter Rule

Always use current_user() in Genie row filters. Using is_member() evaluates the execution service's group membership, not the user's — data leaks silently without any 403.

The Scope Bug

The App integration UI only shows dashboards.genie. The Genie API checks for both genie and dashboards.genie. Missing genie returns:

HTTP 403
"required scopes: genie"

Fix: patch via account-level API since the UI won't show the genie scope option.

Thread Management

  • Use conversation_id for follow-up questions in the same thread
  • New conversation for topic changes — Genie maintains SQL context within a thread
  • Scope patch: PATCH /api/2.0/preview/accounts/{acct_id}/oidc/custom-app-integration/{id} with {"scopes": ["dashboards.genie","genie","sql"]}
Vector Search

Vector Search

Semantic similarity over governed data. No native OBO — per-user filtering is application responsibility.

Identity Model

  • M2M only — App SP has SELECT on the index
  • WorkspaceClient() no-args picks up auto-injected SP creds
  • Scope: vector-search
  • Delta Sync index: auto-updates when source table changes

No native per-user context. Vector Search does not support OBO. The SP retrieves results; your code enforces per-user visibility.

Per-User Filtering Strategies

Pre-filter Include user_group column in indexed data. Pass as filter in similarity query. Faster — only matching rows are scored. Requires group info in index.
Post-filter Retrieve top-K results, then check each against user permissions. Flexible for complex/external ACLs. Wastes retrieval budget on filtered results.
Shared KB M2M is correct — same docs for all users. No per-user filtering needed.
UC Functions

Unity Catalog Functions

The safe pattern for agent write access. Agents get EXECUTE on the function — never raw table grants.

Identity Model

  • Inherits caller's context: OBO → human, M2M → SP
  • Grant required: EXECUTE on the function
  • current_user() inside the function body = whoever triggered SQL
  • Agent SP needs EXECUTE, not SELECT/INSERT/UPDATE on the underlying table

The function enforces business rules and checks current_user() internally. The agent never touches the table directly.

CREATE FUNCTION prod.tools.approve_deal(
  deal_id STRING,
  approver STRING
)
RETURNS STRING LANGUAGE SQL
RETURN (
  UPDATE prod.sales.deals
  SET
    status      = 'approved',
    approved_by = approver
  WHERE id = deal_id
    AND current_user() IN (
      SELECT email
      FROM prod.sales.approvers
    )
);

Tables & Lakebase

Tables & Lakebase

Row filters and column masks for standard tables. Lakebase is the only resource that does not use UC for authorization.

Tables — Row Filters & Column Masks

  • SQL function evaluated at query time, attached to the table
  • Fires regardless of access path: notebook, API, Genie, agent
  • Use current_user() — correct in every access path
  • Column masks: transform column values per-user, same trigger rules
  • ABAC with governed tags: tag columns with sensitivity, filter on user attributes

Lakebase — PG-Native (not UC)

The only resource that uses PostgreSQL-native authorization: PG roles, GRANT, and RLS policies. UC does not govern direct PG connections.

  • AuthN: Databricks OAuth token as PG password, or native PG roles
  • App's SP client ID becomes the PG role name — auto-created on resource binding
  • Role gets CONNECT + CREATE; additional grants (SELECT, INSERT) added manually
  • Token refresh: SDK generate_database_credential() or @databricks/lakebase npm pkg
NOT Unity Catalog

Two AuthZ Paths for Same Data

Direct PG connection (apps, psql): PG roles + GRANT + RLS govern access.

Via lakehouse / DBSQL: UC policies apply, not PG grants.

Same data, different auth model depending on access path.

UC HTTP Connections

UC HTTP Connections

The governed boundary for every external API call. Credentials live in UC, never in code.

What It Provides

  • Governed access to external APIs: Jira, GitHub, Slack, Salesforce, etc.
  • Credentials stored in UC — never in environment variables or source code
  • GRANT USE CONNECTION ON CONNECTION <name> TO <principal>
  • REVOKE removes access immediately — no redeploy needed
  • Platform injects auth headers; agent code is credential-free

Runtime kill switch. Revoke USE CONNECTION and the agent loses the tool immediately. The most granular access control for external services.

External MCP via Connections

Third-party MCP servers (Jira, GitHub, Slack) accessed through UC HTTP Connections. IDE integration via uc-mcp-proxy for Cursor/Windsurf. Managed OAuth providers: Glean, GitHub, Atlassian.

Four Auth Methods — Two-Sided Identity

MethodExternal service sees
Bearer TokenShared (one static token)
OAuth M2MShared (app client credentials)
OAuth U2M SharedShared (one user's delegated token)

Managed OAuthPer-user identity ✓

Only U2M Per User and Managed OAuth give true per-user identity on both the Databricks side and the external service side.

No Read-Only Scope at Connection Level

UC controls who can use the connection, not what they can do through it. Create separate connections with different external credentials (read-only vs. read-write) and GRANT to different groups.

Agent Bricks

Supervisor / Sub-agent Orchestration

How Agent Bricks routes intent, manages identity scope, and composes multi-agent workflows.

Supervisor Model

  • Coordinator deployed as a Model Serving endpoint
  • Up to 20 sub-agents per supervisor
  • Sub-agent types: Knowledge Assistant, Genie Space, Custom Tool (any Model Serving endpoint)
  • Supervisor routes user intent to the right sub-agent; sub-agents do not communicate directly

OBO Mode — Full User Propagation

Supervisor auto-forwards the user's token to all sub-agents. current_user() = human email from supervisor through to the deepest tool call. Row filters, Genie RLS, and function auth all see the actual human.

M2M Mode: Supervisor uses SP credentials. Sub-agents see the SP UUID. Use when background processing or shared data access is intentional — not for per-user data isolation.

Critical Limits & Scope Requirements

  • Max 20 sub-agents per supervisor chain
  • Each sub-agent's tools need their own UC grants — grants don't cascade from supervisor
  • Genie sub-agents need both genie and dashboards.genie scopes
  • Missing scope → 403: "required scopes: genie"
  • Grant UC permissions to the SP of the sub-agent, not the human user

MCP

MCP — Three Types & Tool Selection

How to pick the right MCP type, and which tool a multi-tool agent calls when names conflict.
Managed

Databricks-Hosted

Pre-built connectors (Unity Catalog). No code, no infra. Built-in UC permissions and audit. Governed by default.

Auth: Platform-managed
Use when: You need UC catalog browsing, table discovery, or standard Databricks platform tools without custom logic.
Custom

Databricks Apps

Your code deployed as a Databricks App (Streamlit/FastAPI). Full control over tools, auth, and business logic.

Auth: OBO, M2M, or mixed
Identity: App proxy injects X-Forwarded-Email (proxy-signed). Use OBO for per-user data access, M2M for shared operations.
Use when: You have proprietary business logic or need custom auth flows.
External

Via UC Connections

Third-party MCP servers accessed through UC HTTP Connections. Credential-free code; platform injects auth.

Auth: GRANT USE CONNECTION
Examples: Jira, GitHub, Slack, Salesforce
Use when: Third-party SaaS integrations where you want governed, auditable, revocable access.

Tool Selection Priority (name collision)

When an agent has access to multiple MCP servers with overlapping tool names: 1) Managed (highest trust) → 2) Custom (explicit business logic) → 3) External (third-party). Configure explicit tool routing in the supervisor when ambiguity exists.

SDK Integration

OpenAI Agents SDK: native MCP client. LangGraph: MCP tools as nodes. uc-mcp-proxy: local bridge for IDE clients (Cursor, Windsurf) to Databricks-governed connections.

Token Federation

Token Federation in Multi-Agent Context

When agents span organizational boundaries or external IdPs, federation eliminates cross-org credential sharing.

When Federation is Needed

  • Cross-IdP orchestration: an agent running under your Entra ID needs to call a Databricks-governed resource without a Databricks PAT
  • CI/CD pipelines: GitHub Actions or AWS IAM workloads that need to act as a Databricks SP — no secrets in workflow config
  • External partners: a partner org's IdP token exchanged for scoped Databricks access
  • Multi-agent handoffs: agent A (external IdP) chains to agent B (Databricks) without sharing secrets

The Exchange

POST /oidc/v1/token
grant_type=urn:ietf:params:
  oauth:grant-type:token-exchange
subject_token=<idp-jwt>
subject_token_type=
  urn:ietf:params:oauth:token-type:jwt
requested_token_type=
  urn:ietf:params:oauth:token-type:
  access_token

Account Token Federation

  • Users + SPs from your IdP (Entra, Okta, PingOne, etc.)
  • SCIM required — users must be synced first
  • Interactive U2M + M2M flows
  • Refresh tokens for long-lived sessions
  • Up to 5 issuers

Workload Identity Federation (WIF)

  • Workload → SP only (M2M)
  • No SCIM required
  • Unlimited issuers — each SP has its own policy
  • AWS IAM and GitHub Actions OIDC: WIF only
  • Secretless: no Databricks credentials in external infra

TTL: Databricks copies the IdP's exp claim verbatim. Short-lived IdP tokens = short-lived Databricks tokens. Plan token refresh in agent code.

Unity AI Gateway

Unity AI Gateway in Orchestration

One governance surface for LLM calls, MCP tool invocations, and coding agents. See Deck 05 for full deep dive.

What It Governs

  • Model traffic: every LLM call from agents, coding agents (Cursor, Codex CLI, Claude Code), and deployed apps
  • MCP calls: external, custom, and managed MCP servers — same permissions, rate limits, and audit as LLM endpoints
  • Guardrails: PII redaction/block, unsafe content, jailbreak, hallucination, custom LLM-judge — evaluated on every request and response
  • Rate limits: per endpoint, per user, per group, per SP — QPM and TPM
  • Traffic splitting / fallback routing: A/B model testing, provider failover without client changes

Identity Through the Gateway

  • Each coding agent gets its own OpenAI-compatible base URL — developer's Databricks identity flows into system.ai_gateway.usage
  • App agents route LLM calls via gateway endpoint — SP identity attributed in audit and cost
  • Rate limits enforced per principal: a runaway agent is throttled without affecting other users

Observability

  • system.ai_gateway.usage: every call, every principal, every tag — join with system.billing.usage for cost attribution
  • Inference tables: request/response payloads to UC Delta for audit and offline eval
  • system.access.audit: join on request_id for end-to-end agent lineage

See Deck 05 (06-unity-ai-gateway-capabilities.html) for full deep dive: guardrail configuration, traffic splitting mechanics, coding agent setup, and inference table schema.

Decision Guide

Which Token Path?

Three questions that determine token path, resource auth model, and connection type.
Question 1

Do users need per-user data isolation at the resource?

Yes Use OBO. Pass user token to resource. Row filters see human email.
No Use M2M. SP identity, shared access, background processing.

Vector Search always M2M. Apply pre/post filter in your code for per-user results.

Question 2

Does the user need per-user identity at the external service?

Yes UC Connection with U2M Per User or Managed OAuth.
No Bearer token or OAuth M2M connection is sufficient.

Only U2M Per User and Managed OAuth give per-user identity on both sides of the connection.

Question 3

Does the agent need to write to a table?

Yes Wrap in UC Function. Grant EXECUTE to SP, not table-level INSERT/UPDATE.
No Row filters + column masks on the table. current_user() only in filter functions.

UC Functions are the authorization boundary for write operations. Principle of least privilege.

Lakebase: PG GRANT + RLS regardless of token path. Auth model differs between direct PG connection and lakehouse/DBSQL federation. Same data, different governance.

Pre-Production

Production Readiness

Identity & Scopes

  • One SP per agent capability boundary — no shared SPs
  • OAuth scopes minimal: sql, genie, dashboards.genie, model-serving, vector-search only as needed
  • Genie: both genie + dashboards.genie scopes patched via account API
  • Token federation for CI/CD: no Databricks secrets in pipeline config

Data Authorization

  • Row filters: current_user() only — never is_member()
  • Agent writes: via UC Function with EXECUTE, not direct table grants
  • Lakebase: PG GRANT + RLS per table; do not rely on UC for direct PG access
  • Vector Search: pre-filter or post-filter applied — SP retrieval alone is insufficient

External Access

  • Every external API via UC Connection — no credentials in code or env vars
  • USE CONNECTION grants on each connection object per agent SP
  • U2M Per User or Managed OAuth when external service needs per-user identity
  • Test REVOKE: remove connection grant, confirm tool disappears immediately

Observability

  • system.access.audit enabled and retention configured
  • MLflow tracing on all agents — join with audit on request_id
  • AI Gateway usage table wired for cost attribution (usage_context tags)
  • Tokens never appear in log output or exception payloads
