The Problem

Shared identity hides the user

Today (without this pattern)

External app calls Genie using a shared service principal. Unity Catalog evaluates the SP's identity. Row filters are off or return SP-level data. Alice and Bob see the same rows.

With cross-IdP token exchange

Alice's login to your app produces a token that is exchanged for a Databricks token as Alice. Genie runs as Alice. Her row filter fires. She sees only her data. Bob sees only his.

Unity Catalog's RLS and CLS are built on current_user(). The question is never whether enforcement works. It is whose identity reaches the query engine. This pattern ensures it is the human's.

03 / 15

Concept

Two legs. Don't conflate them.

01
App Login
User authenticates to IdP A via your app. Standard OAuth authorization_code flow with PKCE. The app receives an IdP A JWT. This is normal SSO. Databricks is not involved yet.
02
Token Exchange
App sends the IdP A JWT to Databricks /oidc/v1/token using grant type token-exchange (RFC 8693). Databricks validates the JWT against the federation policy and returns a Databricks OAuth token for that user. This token is used for all subsequent API calls.
Leg 1 alone never reaches Databricks. Sending the raw IdP JWT to Genie or SQL endpoints will fail. Leg 2 is mandatory and is where per-user identity enters Databricks.
04 / 15

Gate Zero

How IdP A signs its tokens determines everything

Ask your IdP team this first. The answer decides your integration path.

RS256 / ES256
Token exchange works
Asymmetric signing. Public key is verifiable. JWKS endpoint can be published even if IdP does not serve one natively. Databricks validates the JWT signature before issuing the exchange token.
No JWKS URL
Host the public key yourself
IdP may sign RS256/ES256 but not expose a JWKS endpoint. You can host the public key at any HTTPS URL and configure that as the JWKS in the federation policy. Still works.
HS256
Fallback to SSO login
Shared-secret signing. Databricks cannot validate without the secret. Token exchange is unavailable. Fallback: configure IdP A as SAML/OIDC SSO for the workspace and use Databricks U2M OAuth with a browser redirect.

Start here. Before writing any code or configuring any policy, get the signing algorithm and JWKS URL from the IdP team. Everything else depends on this answer.

05 / 15

Prerequisites

Five things that must be true before the exchange works

1
Users exist in Databricks
SCIM sync from your IdP (or manual provisioning). Federation maps an identity; it does not create one. No SCIM user = exchange returns 401.
2
Federation policy registered for IdP A
Account Console, Security, Federation Policies. Requires: issuer URL, audience, subject claim. One policy per IdP per environment.
3
UC row filters use current_user(), not is_member()
is_member() evaluates the execution service (Genie, Agent), not the calling user. This is the #1 silent RLS failure in OBO deployments.
4
App passes a Databricks user token, not the raw IdP JWT
The IdP JWT is the input to the exchange. The output Databricks token is what goes in the API Authorization header.
5
Receiving endpoint supports OBO extraction
Genie: Authorization: Bearer {user_token} with genie + dashboards.genie scopes. SQL API: Authorization: Bearer {user_token} with sql scope.

IdP A Side

What you need from the external identity provider

What you need Where it goes Why it matters
JWKS endpoint URL required https://idp-a.example.com/.well-known/jwks.json Databricks fetches public keys to validate JWT signatures. Without this, token exchange cannot work.
Issuer (iss) value https://idp-a.example.com/ Must exactly match the iss claim in issued JWTs. Even a trailing slash difference causes 400.
Audience (aud) value api://my-app-id or custom value Must match the aud claim in JWTs. If it differs per environment, you need one federation policy per environment.
Subject claim name email, upn, or sub The claim that holds the user's identity. Its value must exactly match the user's email in Databricks. Must be immutable.
Signing algorithm confirmation RS256 or ES256 HS256 means token exchange is not possible. Determines whether you can proceed with this pattern.
07 / 15

Databricks Side

Account-level federation policy setup

Federation Policy

Account Console → Settings → Security → Authentication → Federation Policies → Add

  • Issuer URL — IdP A's OIDC issuer. Databricks fetches .well-known/openid-configuration or the JWKS URL directly.
  • Audience — must exactly match the aud claim in IdP A's tokens.
  • Subject claim — the claim whose value maps to the user's Databricks account email (e.g. email, upn).
  • Scope — account-wide (applies to all users). No client_id = per-user token. Including client_id switches to SP token.

SCIM / User Provisioning

Account Console → User Management → SCIM token → configure in IdP A

  • All users who will exchange tokens must be provisioned to the Databricks account first.
  • SCIM sync from IdP A (or from your workspace SSO IdP) is the standard approach.
  • Groups should also be synced if row filters or column masks use is_account_group_member().
  • Provisioning must complete before the first exchange attempt.
Account limit: up to 5 federated token issuers per Databricks account. If you have more IdPs, consolidate or use the SSO fallback for the extras.

Under the Hood

The token exchange: one POST, no secrets

POST https://<workspace>.azuredatabricks.net/oidc/v1/token Content-Type: application/x-www-form-urlencoded grant_type=urn:ietf:params:oauth:grant-type:token-exchange subject_token=<IdP A JWT received from login> subject_token_type=urn:ietf:params:oauth:token-type:jwt scope=all-apis # or: sql, genie, dashboards.genie # Response: { "access_token": "<databricks-oauth-token>", "expires_in": 3600 } # Use the access_token as Bearer in all Databricks API calls # WARNING: do NOT include client_id — that switches to a service principal token
✓

No Databricks client secret. The IdP JWT is the credential. No secrets to rotate or store in the app.

!

Always exchange a fresh IdP JWT. Databricks copies the JWT's exp verbatim. A stale token produces a Databricks token that expires immediately.

✓

Works for all Databricks APIs. SQL statements, Genie, Model Serving, Vector Search, Databricks SDK.

!

No client_id. Including client_id selects a service principal, not the user. Per-user RLS breaks silently.

10 / 15

Unity Catalog

current_user() enforces per-user access. is_member() does not.

Correct: current_user()
CREATE FUNCTION mask_salary(val DOUBLE) RETURNS DOUBLE RETURN IF(current_user() IN ( SELECT email FROM hr.approved_viewers), val, NULL);
current_user() evaluates the OBO caller's identity. When Alice calls Genie, which calls SQL as Alice, current_user() = alice@company.com. Row filter fires for Alice's rows only. Works in all OBO scenarios.
Wrong: is_member() in OBO
CREATE FUNCTION mask_salary(val DOUBLE) RETURNS DOUBLE RETURN IF(is_member('hr-team'), val, NULL);
is_member() evaluates the execution context, not the calling user. In Genie or Agent Bricks, the execution context is the Genie service identity, not Alice. The filter fires for the wrong identity. All OBO users see the same result.
Universal rule: Use current_user() in all row filters and column masks. It is the only function that correctly identifies the OBO caller in every execution path.

End User

What the user experiences: nothing new

1
User logs in to the external app Standard SSO via IdP A. No Databricks login screen, no separate credentials. Business as usual.
2
App silently exchanges token invisible Single POST to /oidc/v1/token. Completes in under 100ms. User sees nothing.
3
App calls Genie or SQL as the user invisible Databricks user token in the Authorization header. User still sees nothing different.
4
User receives their data Scoped to their identity by Unity Catalog. Row filters and column masks have already fired. No extra steps for the user.
The token exchange happens entirely server-side in the application layer. Users are not aware of it. Re-exchange happens transparently when the token nears expiry.

Operational Risks

What will go wrong if you miss it

Including client_id in the exchange
Silently produces a service principal token, not a user token. Row filters fire for the SP. No error. Per-user RLS breaks. Omit client_id for user-level exchange.
Audience mismatch (400 invalid_grant)
The aud claim in the JWT must exactly match the federation policy. Different per environment (dev/stage/prod)? One policy per environment or use a shared audience value.
Using is_member() in row filters
Evaluates the execution service identity (Genie, Agent Bricks), not the calling user. All OBO users see the same result. Use current_user() exclusively.
Token TTL: exp is copied verbatim
Databricks copies the IdP JWT's exp field exactly. Exchanging a token with 5 seconds remaining produces a Databricks token that expires in 5 seconds. Always exchange a fresh IdP token.
User not in Databricks before exchange
Federation maps an identity; it does not create one. SCIM sync must run before the first exchange attempt for that user. Provision first, then enable the exchange.
5-issuer account limit
Databricks allows up to 5 federated token issuers per account. More IdPs require consolidation or the browser-redirect SSO fallback for the extras.

Action

Questions for your IdP team

A. Token Signing
1
What algorithm signs your tokens? RS256, ES256, or HS256? This single answer determines whether token exchange is possible.
2
Do you expose a public JWKS URL or OIDC discovery endpoint? If not, can you share the public key?
B. Issuer and Claims
3
What is the token issuer (iss) value? It must match exactly.
4
What aud value do tokens carry? Does it differ per environment?
C. Subject and Provisioning
5
Which claim holds the user identity (sub, email, upn)? Its value must match the user's email in Databricks. Must be immutable.
6
Can you provision users to Databricks via SCIM? Can groups be synced too?
D. Fallback Readiness
7
Can you act as a SAML 2.0 or OIDC SSO provider? (Needed if signing algorithm is HS256.)
8
What is the token lifetime? Is there a refresh mechanism?
Start with question 1. It alone unblocks the rest of the design. The Slack template with all eight questions is in the fieldkit at auth/peruser-byoidp-federation.md.
