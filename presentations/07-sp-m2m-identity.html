Service Principal Identity
& M2M Governance on Databricks

Scoping service-principal credentials to service, group, and Unity Catalog privilege, using the client-credentials (M2M) flow.

Nightly ETL · Customer API Gateway · Compliance Reporting — each with its own SP, its own group, its own privilege scope

Deck 07 — companion to Deck 05 (Human OBO Identity). Deck 05 = human calls Databricks as themselves. This deck = service calls Databricks as a controlled identity.

Agenda

What this deck covers

The problem

One shared service-principal credential used across multiple services — and the blast-radius, rotation, and audit problems that design creates.

What

The SP → Group → Unity Catalog privilege chain, and the client-credentials (M2M) token flow that underlies it.

Why

Blast-radius reduction, clean audit attribution, and independent credential rotation — concrete risk and cost outcomes.

How

Step-by-step setup, secret management, token scopes, and the SP-vs-OBO decision gate for choosing the right identity model.

The issue

One SP for everything is one breach away from everything.

Acme started with a single service principal. All three services used it. It had admin-level Unity Catalog grants because “it’s easier.” Then it leaked in a config file.
3
services on 1 secret
1
rotation disrupts all

What rotation looks like without scoped SPs

Update the secret → redeploy the ETL pipeline → redeploy the API gateway → redeploy the compliance job → coordinate the change window → something breaks. Every. Time.

The three Acme failure modes

Over-privilege

The compliance job only needs SELECT on audit tables. The shared SP has MODIFY on production bronze. Now the compliance job can corrupt data.

Opaque audit trail

system.access.audit shows one SP doing everything. Was it the ETL job? The API gateway? The compliance run? Impossible to tell.

Blast radius on compromise

One leaked secret grants access to every catalog, every table, every model endpoint the SP was ever granted access to. Revoke it and all three services go down simultaneously.

What: token acquisition

The client credentials flow

An SP gets a token by posting its identity to the Databricks OIDC token endpoint. No human in the loop.

HTTP request (M2M)
POST https://<workspace>/oidc/v1/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id=<application-id>
&client_secret=<secret-value>
&scope=all-apis
Token response
{
  "access_token": "eyJhbG...",  // JWT
  "token_type":   "Bearer",
  "expires_in":   3600           // seconds
}

The token is a standard Bearer JWT. Pass it in the Authorization header on every Databricks API or SQL call. The SP identity rides inside the JWT—no session state on the server side.

Contrast: Deck 05 human OBO flow

Human OBO (Deck 05)

  • User authenticates via IdP (browser redirect or device code)
  • Token exchange propagates the human’s identity downstream
  • UC row filters evaluate the human’s group membership
  • current_user() returns a human email address
  • Audit shows the human’s identity on every action

Service Principal M2M (this deck)

  • SP authenticates via client credentials—no browser, no human
  • Token is scoped to the SP’s configured scope (e.g. all-apis or sql)
  • UC row filters evaluate the SP’s group membership
  • current_user() returns the SP’s application ID—a UUID, not an email
  • Audit shows the SP’s identity on every action

Key difference: current_user() for an SP returns its UUID-format application ID, not a human email. Row filters and column masks work identically for both—the evaluation mechanism is the same, only the identity token differs.

What: identity design

One SP per service. One credential to rotate.

If two services share a credential, you lose the ability to rotate independently. Scope by function, not by team.

Independent rotation: ETL secret rotates without touching the API gateway or compliance job

Least privilege by design: compliance job cannot write bronze data even if compromised

Clear audit attribution: every action in system.access.audit maps to exactly one service

Why it matters: least privilege

Scope is a second layer of least privilege on top of UC grants.

A token scope limits what API surface the SP can reach—even if it has UC grants on a broader set of resources.

all-apis

ETL pipeline

Full Databricks REST API surface: Jobs, DBFS, Clusters, SQL, Secrets. Needed because the ETL pipeline starts jobs, writes to DBFS staging, and runs SQL transforms.

Use only when the service genuinely needs both compute and SQL APIs.

sql

API gateway SP

SQL warehouse access only. The gateway serves predictions from a pre-built silver table via SQL. It does not need to start jobs, manage clusters, or read secrets.

Preferred scope for read-only serving patterns. Cannot call Jobs or Cluster APIs even if misconfigured.

sql

Compliance SP

SQL warehouse access for SELECT on system.access.audit. Reports are generated by reading from the system table and writing output to an external report store.

Even if the SP secret leaks, the attacker cannot start a cluster or modify jobs with a sql-scoped token.

Layered defense: UC group membership controls which data the SP can read. Token scope controls which API surface the SP can reach. Compromise one layer and the other still limits the blast radius.

Why it matters: cost & risk governance

When the API gateway SP calls a model, the SP is the rate-limit subject.

The same governance machinery from Deck 07 applies to SP callers. The SP’s application ID is the identity Unity AI Gateway tracks.

Per-SP rate limit config

Unity AI Gateway identifies the caller by the SP’s application ID in the JWT. Rate limits and spend caps apply at that identity level—not at the service-name level, which is mutable.

Spend cap per SP

Set a daily spend cap on api-gateway-sp. If the gateway has a runaway loop, the cap fires before the bill does. Other SPs are unaffected.

Audit in system.ai_gateway.usage
-- SP traffic through Unity AI Gateway
SELECT
  timestamp,
  client_id,   -- SP application ID (UUID)
  model_name,
  input_tokens,
  output_tokens,
  latency_ms
FROM system.ai_gateway.usage
WHERE client_id = '<api-gateway-sp-app-id>'
  AND timestamp >= now() - interval 7 days
ORDER BY timestamp DESC;

Complete SP visibility: every model call by api-gateway-sp appears in system.ai_gateway.usage with token counts, latency, and the SP’s identity. No custom instrumentation required.

How: permission boundaries

Groups are the unit of governance—not SPs.

Grant to the group, not the SP
-- Grant privilege to the group
GRANT SELECT ON CATALOG prod
  TO GROUP pipeline-readers;

-- Add the SP to the group (one line)
-- Done via Databricks Groups API or UI
-- No GRANT statement needed

-- Remove SP from group = instant cutoff
-- No REVOKE statement needed
-- No redeployment of the service

Adding a second pipeline SP

A new batch job needs the same bronze access. Add batch-loader-sp to pipeline-readers. One API call. No new GRANT statements. Audit shows a separate identity.

Audit group membership
-- Who is in each SP group right now?
SELECT
  group_name,
  member_name,
  member_type,
  change_time
FROM system.access.group_membership
WHERE group_name IN (
  'pipeline-readers',
  'api-readers',
  'compliance-readers'
)
ORDER BY change_time DESC;

Group membership is a system table—queryable, joinable, alertable. You can build a Databricks SQL dashboard that shows every SP group and its current members without leaving the platform.

How: runtime enforcement

Unity Catalog enforces SP identity the same way it enforces human identity.

What UC sees when an SP queries
-- current_user() for a service principal
-- returns the SP's application ID (UUID)
-- NOT a human email address
SELECT current_user();
-- returns: "a1b2c3d4-e5f6-7890-abcd-ef1234567890"

-- is_member() evaluates SP's group membership
-- works identically for SPs and humans
SELECT is_member('compliance-readers');
-- returns: true  (if SP is in the group)

Key call-out: current_user() returns the SP’s UUID-format application ID when called by an SP. Build dashboards and audit queries that handle both email-format and UUID-format identities.

Row-level security using SP group membership
-- Row filter on audit table: compliance SP
-- only sees rows from its own region
CREATE FUNCTION compliance_region_filter(
  region STRING
)
RETURNS BOOLEAN
RETURN
  is_member('compliance-readers')
  AND region = 'us-east-1'
  OR is_member('compliance-global');

ALTER TABLE system.access.audit
  SET ROW FILTER compliance_region_filter
  ON (region);

Column masks apply identically. An SP calling a model endpoint via SQL goes through the same RLS/CLS evaluation path as a human analyst running the same query.

How: credential storage

Where the secret lives determines your blast radius.

Where NOT to put the client secret

Plaintext environment variables

Visible in process listings, container environment dumps, and CI logs. One printenv away from a breach.

Code repositories

Even “private” repos are scanned by credential detection tools. One accidental fork or public push and the secret is permanently in git history.

Shared config files

A config file on a shared NFS or S3 bucket grants implicit access to every service and person who can read the bucket. This is a shared credential problem in disguise.

Databricks secret scope (backed by your cloud KMS)
# Create a secret scope (backed by cloud KMS)
databricks secrets create-scope \
  --scope acme-etl-secrets

# Store the SP client secret
databricks secrets put-secret \
  --scope acme-etl-secrets \
  --key etl-sp-secret \
  --string-value <secret-value>
Reference in job config (never the value)
{
  "env_vars": {
    "DATABRICKS_CLIENT_SECRET":
      "{{secrets/acme-etl-secrets/etl-sp-secret}}"
  }
}

Secret scope values are never logged, never returned by the API. The {{secrets/...}} reference is resolved at runtime inside the Databricks execution environment. Backed by Azure Key Vault, AWS Secrets Manager, or GCP Secret Manager.

How: full audit visibility

Three queries cover the complete SP audit story.

1 — SP data access (last 7 days)
-- All data actions by compliance-sp
SELECT
  event_time,
  action_name,
  request_params.full_name_arg
    AS table_accessed,
  source_ip_address
FROM system.access.audit
WHERE
  user_identity.email
    = '<compliance-sp-app-id>'
  AND event_time
    >= now() - interval 7 days
  AND service_name = 'unityCatalog'
ORDER BY event_time DESC;
2 — Group membership changes
-- Who was added or removed from SP groups?
SELECT
  change_time,
  group_name,
  member_name,
  change_type  -- ADD or REMOVE
FROM
  system.access.group_membership_changes
WHERE group_name IN (
  'pipeline-readers',
  'api-readers',
  'compliance-readers'
)
ORDER BY change_time DESC;
3 — Token issuance events
-- SP token issuance (auth events)
SELECT
  event_time,
  user_identity.email
    AS sp_identity,
  action_name,
  source_ip_address,
  request_params.scope
FROM system.access.audit
WHERE
  service_name = 'accounts'
  AND action_name
    = 'tokenLogin'
  AND event_time
    >= now() - interval 1 day
ORDER BY event_time DESC;

Zero custom instrumentation: the audit trail is complete from day one. Group membership, data access, and token issuance are all in system tables, queryable with standard SQL, alertable via Databricks SQL dashboards.

How: decision gate

Use SP when no human is in the loop. Use OBO when the human matters.

The choice is about identity semantics, not convenience. When downstream RLS must enforce per-human access, the SP cannot substitute.

Use SP (this deck)

Background job — nightly ETL

Runs on a schedule, no human triggers it, no human-specific row access required. The SP identity is the right semantic: the pipeline speaks for itself, not for any individual engineer.

Use OBO (Deck 05)

Interactive query — analyst explores data

The analyst triggers the query, and UC row filters must evaluate that analyst’s group membership. An SP here would bypass per-human RLS: all analysts would see the same data as the SP’s group allows.

Use SP (this deck)

Automated report — compliance job

Generates a regulatory report on a weekly schedule. The report is authored by the compliance process, not by any individual. Audit should show the SP identity, not the engineer who deployed it.

Use OBO (Deck 05)

User-triggered export — self-service download

A product manager clicks “Export my team’s data.” The UC row filter must enforce that they only export their own team’s rows. The call must carry the user’s identity downstream via token exchange.

How: implementation

Eight steps from zero to governed M2M identity.

1
Create one SP per service

In the Databricks account console: one SP for ETL, one for the API gateway, one for compliance. Name them by function, not by team or environment.

2
Create one Databricks group per SP

pipeline-readers, api-readers, compliance-readers. The group is the unit of governance—the SP is just a member.

3
Grant UC privileges to the group, not to the SP

GRANT SELECT ON CATALOG prod TO GROUP pipeline-readers. Never grant directly to the SP’s application ID.

4
Add each SP to its group

Via the Databricks Groups API or account console. The SP now inherits all group privileges without any additional GRANT statements.

5
Store the client secret in a Databricks secret scope

Never in environment variables, config files, or source code. Reference it in job configs as {{secrets/scope-name/key-name}}.

6
Configure the token scope to the minimum needed

sql for read-only services. all-apis only for services that must start jobs or manage compute.

7
Add the SP’s group to UC row filters and column masks if RLS is required

Use is_member('group-name') in row filter functions. No SP-specific code—the filter works for any member of the group, human or SP.

8
Verify the audit trail in system tables

Run the three queries from Slide 12: SP data access, group membership changes, token issuance events. If all three return expected rows, the identity chain is working end to end.
