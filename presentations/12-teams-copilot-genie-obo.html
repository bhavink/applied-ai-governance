The issue

Genie and Agent Bricks are reachable from Teams and Copilot Studio through four different paths — each handling identity differently.

4
distinct integration paths into Genie from Teams and Copilot Studio

The four paths, by name

  • Custom AI agent + Azure Bot Service + OAuth federation
  • Copilot Studio via MCP, “End user credentials”
  • Copilot Studio via MCP, “Maker credentials”
  • Native “Databricks Genie” app (Teams / M365 Copilot marketplace)

The question every rollout has to answer

When two different employees ask Genie the same question from inside Teams or Copilot Studio, do they get answers scoped to their own row-level permissions — or the same answer regardless of who's asking?

The answer depends on which of the four paths is in use, and on two separate settings, not one. The four paths look similar from the client side, which is exactly what makes this easy to get wrong.

What — two gates, not one

Per-user identity depends on two separate settings.

Gate 1 — Token layer

Did a real per-user token reach Databricks?

OAuth federation, no client_id in the exchange, “End user credentials” selected in Copilot Studio. This is the gate everyone checks — and the one all the setup guides walk you through.

Gate 2 — Genie space publish mode

Is the space published VIEWER, not EMBEDDED_CREDENTIALS?

This lives on the space, not the client integration. It's invisible from the Teams bot's or Copilot Studio connector's configuration screen — you have to check the space directly.

Why this matters

Every Teams and Copilot rollout hits this question — and getting it wrong fails silently.

“We're putting Genie in Teams so people don't have to leave their chat window. But when Priya and Sam both ask ‘what's my region's revenue,’ do they get different answers — or the same one?”

The assumption that trips people up

“We used OAuth / OBO, so we're covered.” Getting the token layer right is necessary — and, on its own, not sufficient. There's a second setting, on the Genie space itself, that decides whether any of that per-user auth actually reaches the data.

The concrete consequence

If Gate 2 is missed, every user sees the exact same data — regardless of how correctly the token exchange, OAuth federation, or Copilot Studio credential setting is configured upstream. There's no error and no warning; the space just quietly runs every query as its publisher.

  urn:ietf:params:oauth:token-type:jwt

This is the identical exchange already documented for external apps with a bring-your-own IdP. Teams/Azure Bot Service is a new front-door, not a new mechanism.

How — check the publish mode

One setting decides who the SQL runs as — regardless of the token.

Publish modeWho the SQL runs asPer-user RLS?
VIEWER The calling identity — the human, if a per-user token reached the API Yes
EMBEDDED_CREDENTIALS The space's publisher — always No

Where to check: the Genie space's own settings — not the Teams bot configuration and not the Copilot Studio connector. Gate 2 lives entirely on the space, so a review of the client integration alone will never surface it.

How — what to do, per path

Which path, and what to verify on each.

Building a custom Teams bot

Use Azure Bot Service + OAuth federation

Officially documented pattern. Check Gate 2 on the target Genie space before shipping.

Using Copilot Studio

Select “End user credentials,” not “Maker credentials”

Gives real per-user identity — still gated by Gate 2. “Maker credentials” is fine when shared access is the actual intent.

Already using the native marketplace app

Verify before you promise per-user RLS to anyone

Run the two-user test yourself and confirm the result before this shows up in a customer conversation — standard due diligence for any new integration surface.

Every path, no exceptions

Check the Genie space's publish mode directly

Don't infer it from the client integration's settings. It's a property of the space, and it silently overrides everything upstream of it.

How — gotchas

What breaks quietly if you skip the fine print.
