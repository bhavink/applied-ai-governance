Unity AI Gateway

One surface for governing every AI call your org makes.

LLMs · MCPs · coding agents · Databricks Apps agents · custom external APIs — under one control point.

The answer

Unity AI Gateway: one gateway, three pillars.

Every AI call — from a developer’s coding tool, from a production agent, to an LLM, to a tool server, to an external API — routes through the same control point. Same permissions, same audit, same cost surface.

Pillar 1

LLMs

Agents, LLM endpoints, coding agents (Cursor, Codex CLI, Gemini CLI, Claude Code), and Databricks Apps agents — all governed under one surface, with native multi-provider APIs.

Pillar 2

MCPs

External, custom (Databricks Apps), and Databricks-managed MCP servers under the same permissions, audit, and access controls as LLMs.

Pillar 3

Model Serving Endpoints

Gateway features layered onto existing model serving endpoints — safety, PII, keyword guardrails, and the established system table observability.

One control point. Permissions · rate limits · guardrails · usage tracking · inference logging · cost attribution · fallback routing · traffic splitting.

Pillar 1 · LLMs

Shape LLM Traffic

Four controls the gateway applies to every LLM call — before the request reaches a model and before the response reaches a caller.

Rate Limits

Per endpoint, per user, per group, per service principal. Both call rate (QPM) and token rate (TPM). Cap a runaway agent. Fair-share a model across teams.

Guardrails

Six LLM-based templates evaluate every request and response. Block, sanitize, or alert — configurable per phase.

Traffic Splitting

Distribute requests across model backends. A/B test a new model on 20% of traffic. Stage a gradual rollout from one provider to another with built-in metrics for comparison.

Fallback Routing

Primary provider fails → gateway routes to a secondary served entity. Multi-provider failover (OpenAI → Anthropic → FMAPI) without client code changes.

Pillar 1 · LLMs · Guardrails

Six Guardrail Templates

An LLM evaluator (recommended: databricks-gpt-5-nano) scores every request and response against one of six templates. Configure per phase (input/output), per action (block/sanitize/alert), per mode (sync/async).

TemplateActionWhat it catches
PII RedactionSanitizeReplace PII with placeholders (e.g., [EMAIL], [SSN]) before the model sees the request or the user sees the response.
PII BlockBlockRefuse the request entirely if the payload contains PII. Stricter alternative to redaction for regulated workloads.
Unsafe ContentBlockHate speech, harassment, violence, self-harm, sexual content, weapons, extremism. Both input and output phases.
JailbreakBlockInstruction overrides, obfuscated payloads, role-play exploits, prompt extraction attempts. Input phase.
HallucinationBlock / AlertOutput unsupported by retrieved context. Useful for RAG systems where grounding matters.
CustomYou chooseYour own evaluator prompt — codify domain-specific policies (no internal codenames, no competitor mentions, etc.).

Fail-closed by default. A blocked request returns HTTP 400 with Request blocked by input guardrail '<name>'. Audit trail logs every evaluation, every action.

Pillar 1 · LLMs · Multi-Provider

Native APIs for Every Major Provider

Speak each provider’s native protocol — not a lowest-common-denominator translation. Client SDKs work unchanged.

OpenAI Responses

from openai import OpenAI
client = OpenAI(
  base_url=
    "https://<uag>/v1",
  api_key=DATABRICKS_TOKEN,
)
client.responses.create(...)

Anthropic Messages

from anthropic import Anthropic
client = Anthropic(
  base_url=
    "https://<uag>/anthropic",
  api_key=DATABRICKS_TOKEN,
)
client.messages.create(...)

Gemini

from google import genai
client = genai.Client(
  http_options={"base_url":
    "https://<uag>/gemini/v1"},
)
client.models.generate_content(...)

Why this matters: if an SDK calls a provider-specific feature (streaming events, system prompts in the right shape, structured tool use), it works. No translation layer means no feature drift — and the gateway still applies all the controls.

Pillar 1 · LLMs · Apps Agents

Govern Databricks Apps Agents

Agents authored with Agent Framework and deployed to Databricks Apps route their LLM calls through the gateway. One code change — the agent inherits the gateway’s entire control surface.

Wiring

from openai import OpenAI

# Point at a UAG endpoint
# instead of a serving endpoint
client = OpenAI(
  base_url=(
    "https://<workspace>"
    "/serving-endpoints/<uag>"
  ),
  api_key=os.environ["DATABRICKS_TOKEN"],
)

# All agent LLM calls now flow
# through the gateway
response = client.chat.completions.create(
  model="databricks-meta-llama-3-3-70b",
  messages=[...],
)

The agent inherits

  • Permissions — only principals with access to the UAG endpoint can invoke
  • Rate limits — per-user / per-group QPM and TPM enforcement
  • Guardrails — input/output filtering applies before the call reaches the model
  • Usage tracking — agent identity flows into system.ai_gateway.usage
  • Inference logging — request/response pairs land in UC Delta tables
  • Cost attribution — usage_context tags carry through
Pillar 2 · MCPs

Govern MCPs Through the Same Gateway

Model Context Protocol servers — the tool layer your agents call — sit behind the same control surface as your LLM endpoints. Three modes, one set of primitives.

External MCP

Hosted by: vendor or external infra.

Governed via managed connections — network egress controls, audit logging, access control under the same UAG permissions.

Custom MCP

Hosted by: you, as a Databricks App.

App identity, OBO/M2M auth, UC access controls, payload logging — all flow through the gateway.

Databricks-Managed MCPs

Hosted by: Databricks. Native.

Genie, Vector Search, UC Functions. Built-in UC permissions and audit — no infra to manage, governed by default.

Same primitives across all three: permissions on the server, rate limits per user/group, audit logging per call, optional payload capture. The agent doesn’t know — or care — which mode it’s talking to.

Pillar 1 · LLMs · Custom APIs

Govern Any External API

The gateway isn’t limited to LLM calls. Register an arbitrary external HTTP API and it inherits the same governance — permissions, rate limits, audit logging, payload capture.

Use cases

  • An agent calls a SaaS analytics API
  • A Cursor user hits a private code search service
  • An app talks to a third-party billing endpoint
  • A custom embedding service on a partner cloud

In every case, the external service inherits the gateway’s access controls, rate limits, and audit story — without modifying client code beyond a base URL change.

Why this closes the gap

In the old world, “AI governance” meant governing the model endpoint — and stopping there. The agent could call anything else (a vector DB, a private search index, a customer’s CRM) under whatever credentials, with no audit.

Custom APIs through UAG mean the entire tool surface an agent touches — LLM, MCP, plain HTTP — sits behind one gate.

Pillar 3 · Serving Endpoints

The Layered Path

If your workload is already on a model serving endpoint — an external model, FMAPI pay-per-token, FMAPI provisioned, or a custom model — gateway features layer on with one API call. No re-architecture.

One PUT, full governance

PUT /api/2.0/serving-endpoints/<name>/ai-gateway

{
  "rate_limits": [{"calls": 100, "renewal_period": "minute"}],
  "guardrails": {
    "input": {"safety": true, "pii": {"behavior": "BLOCK"}},
    "output": {"safety": true, "pii": {"behavior": "BLOCK"}}
  },
  "usage_tracking_config": {"enabled": true},
  "inference_table_config": {
    "enabled": true,
    "catalog_name": "main",
    "schema_name": "ai_governance"
  }
}

When to use the layered path

  • You have existing serving endpoints in production — flip on the gateway features without re-pointing clients
  • Your guardrail needs are covered by safety / PII / keyword filters rather than the six LLM-based templates
  • You want classic inference table audit on a custom model endpoint
  • Workload is not on the sidebar path yet — both paths coexist

SQL for finance

SELECT
  g.endpoint_name,
  g.usage_context['team'] AS team,
  SUM(b.usage_quantity * b.list_price)
    AS dollars
FROM system.billing.usage b
JOIN system.ai_gateway.usage g
  ON b.usage_metadata.endpoint_name
   = g.endpoint_name
 AND b.usage_date = g.request_date
WHERE b.sku_name LIKE '%AI_GATEWAY%'
GROUP BY 1, 2
ORDER BY dollars DESC;

Getting it running

From Zero to Governed in 4 Steps

A new gateway endpoint, governing real traffic, with audit and cost visible — in a single sitting.

1. Enable

Account admin enables Unity AI Gateway from the account console Previews page. Workspace-wide flip.

2. Create endpoint

From the AI Gateway sidebar entry, create a new gateway endpoint. Pick a target served entity (FMAPI, external model, custom). Set rate limits and guardrails inline.

3. Grant access

Add users, groups, and service principals to the endpoint’s permission list. Use the same identity model as Unity Catalog.

4. Point clients

Update agents, Cursor, Codex CLI, and external apps to use the gateway base URL. Existing OpenAI/Anthropic/Gemini SDKs work unchanged.

Inference tables and usage tracking are on by default when you create a gateway endpoint — no additional configuration needed to start seeing data.

Production Ready

Pre-Launch Checklist

✓
Endpoint permissions: least privilege
Specific groups, not workspace-wide. Coding agents accessible only to developer groups.
✓
Rate limits per principal
QPM and TPM caps. Tighter for service principals than for human users.
✓
Guardrails enabled on input AND output
PII block + unsafe + jailbreak on input. PII redact + unsafe on output. Custom guardrail for org-specific policy.
✓
Fallback configured for critical endpoints
Multi-entity setup (e.g., 100/0 traffic split) with fallback enabled so provider outages route automatically.
✓
Inference tables in a catalog with retention
Audit-grade UC catalog. Lifecycle policy for payload data. ACLs scoped to security and engineering.
✓
usage_context tagging in client code
Every agent and app passes {"team":"…","app":"…"} so cost attribution lands automatically.
✓
Usage dashboard shared with finance + security
Pre-built AI/BI dashboard published. Standing review cadence.
✓
Alerting on anomalies
Threshold alerts on spend, error rate, guardrail blocks. Routed to oncall.
