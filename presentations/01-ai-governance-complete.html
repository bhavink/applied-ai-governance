Authentication

One Authentication Layer

Databricks has one authorization layer — Unity Catalog. Authentication can come from multiple sources: the workspace IdP for internal users, or external IdPs via token federation for partners and customers. All paths converge on UC, which enforces governance regardless of how the token was obtained.

MCP Services — New

MCP Services Are Now UC Securables

GRANT EXECUTE ON MCP SERVICE replaces the UC connection proxy as the recommended path for external tool access. MCP Services live in the catalog and inherit all UC governance: audit, lineage, ABAC.

Built-in: system.ai.*

Seven managed services with zero setup. No OAuth plumbing required — Databricks handles credentials via managed OAuth providers.

Grant pattern

Grant EXECUTE on catalog.schema.service to a user, group, or SP. Evaluated on every call via UC privilege checks.

system.ai.* built-in services:

Slack GitHub Atlassian Google Drive Google Calendar Gmail SharePoint
Glean, GitHub, Atlassian, Slack now use managed OAuth providers — not http_request(). http_request() is deprecated for these providers.

Service Policies — New Governance Layer

Guardrails vs. Service Policies

These are distinct controls evaluated on every call. Both matter for enterprise AI.

AI Gateway Guardrails

Content governance. What the model is allowed to say. Input/output filtering, PII detection, toxicity, topic restrictions. Applied at the inference layer.

Service Policies

Access governance. Who can call which tools, which MCP Services, with what approval gate. ABAC layer distinct from UC privilege grants.

A user can have UC EXECUTE on an MCP Service and still be blocked by a Service Policy — for example, requiring manager approval before calling a write-capable tool.

Critical Pitfall

current_user() vs is_member()

The #1 silent RLS failure in AI apps. is_member() evaluates the execution service identity, not the calling user, in OBO chains.

Token Pathcurrent_user() returnsis_member() evaluatesSafe for RLS?
U2MHuman emailHuman's groupsSafe
OBO (direct SQL)Human emailHuman's groupsSafe
OBO via Genie / Agent BricksHuman emailGenie SP's groupsBroken
M2MSP identitySP's groupsConsistent
Universal rule

Use current_user() with an allowlist table for row filters. is_member() silently returns the wrong groups in any OBO chain that passes through a service executor (Genie, Agent Bricks, serving endpoint).

Federation

Token Federation

Any app with a trusted IdP JWT can exchange it for a Databricks token via federation. No secrets.

Account Token Federation

Users & SPs. Requires SCIM sync. Hard limit of 5 federated token issuers per account.

Workload Identity Federation

CI/CD pipelines. Per-SP binding. Unlimited issuers. Completely secretless.

Account-level constraint: Maximum 5 federated token issuers per Databricks account. Plan issuer consolidation early for multi-IdP environments.

Coding Agent Governance

Omnigent — Governing AI Coding Agents

Omnigent is Databricks' meta-harness for AI coding agents. It adds governance controls that standard serving infrastructure does not provide.

OS-level Sandbox

Stateful isolation per session. Agent actions are contained — no lateral movement to host or peer sessions.

Stateful Policy Engine

Policies that span a session, not just individual calls. Persistent audit of what the agent did across the entire work session.

Per-session Spend Budget

Hard cost caps per session. Prevents runaway inference costs from recursive or loop-prone coding agents.

Omnigent sits at Enforcement Layer 4 (Execution Boundary). It governs the agent's environment, not just its outputs.

Exception

Lakebase — The Exception

Lakebase uses PG-native authorization (roles, GRANT, RLS), not UC. Treat it as a separate trust domain.

AuthN

Databricks OAuth token as PG password, or native PG roles

AuthZ

PG GRANT + RLS policies. Not Unity Catalog privilege model.

UC Registration

Register Lakebase → read-only UC catalog for cross-source queries without bypassing PG authZ.

Lakehouse Sync

Continuous CDC to Delta via wal2delta. Once in Delta, full UC governance applies.

Agent Pattern

Agent Architecture — Mapped to Seven Layers

  1. Identity (L1) — Agent gets a dedicated Service Principal
  2. Permissions (L2) — SP has explicit UC grants, including EXECUTE on MCP Services — zero default access
  3. Data Governance (L3) — Row filters fire on every data access; BROWSE for catalog discovery
  4. Execution Boundary (L4) — Runs in Model Serving or Apps sandbox; coding agents use Omnigent isolation
  5. Service Policies (L5) — Tool access rules enforced per call; approval gates for sensitive actions
  6. Content Governance (L6) — AI Gateway guardrails on all inference routes
  7. Audit (L7) — Traced via MLflow + system.access.audit; inference tables on agent serving endpoints

Checklist

OBO Prerequisites

  • User exists in Databricks (SCIM sync active)
  • Token federation policy or OAuth app integration configured (5-issuer account limit applies)
  • Row filters use current_user() + allowlist table, never is_member()
  • App passes a valid Databricks user token (not its own SP token)
  • Receiving endpoint supports OBO token extraction
  • Service Policies reviewed for target tools and MCP Services

Security

Security Checklist

  • No static secrets in env vars — MCP Services hold credentials via managed OAuth
  • OAuth scopes enforce least privilege per app
  • UC governance at SQL engine level (row filters, column masks)
  • BROWSE privilege for catalog discovery without data access
  • Service Policies configured for all tool-calling agents
  • AI Gateway guardrails on every inference route
  • Audit via system.access.audit + MLflow traces
  • Inference tables enabled on agent serving endpoints
  • is_member() not used in row filters (replaced with current_user() + allowlist)
  • Tokens never logged; HTTPS enforced at all boundaries

Differentiation

What's Different About This Approach

Identity, not secrets

No PATs, no static credentials. MCP Services + managed OAuth + Federation + OBO everywhere.

Authorization at the resource

UC grants per resource type. MCP Services as UC securables bring tool access into the same model as data access.

Two-tier content + access control

Service Policies govern access. AI Gateway guardrails govern content. Both evaluated on every call, independently.

Audit is automatic. system.access.audit captures every API call. MLflow traces every agent step. Inference tables capture model I/O on agent serving endpoints.
