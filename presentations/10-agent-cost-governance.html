The issue

Agent spend and risk are invisible without governance at the gateway.

An engineer left a coding agent running overnight to refactor a legacy module. The agent looped. Three hours and $400 later, finance flagged the invoice — nobody had visibility while it was happening.

Why this happens

Unity AI Gateway is the chokepoint all agent traffic already passes through. Without governance configured on top of it, it is a pass-through — the gap is configuration, not a limitation of the gateway itself.

Where the spend and risk originate

Autonomous coding agents

Multi-step agentic loops. Can run for hours without a human checkpoint.

IDE completion tools at scale

Many engineers hitting completions constantly adds up faster than it is usually modeled.

Notebook and data-profiling agents

Token-heavy prompts, long context windows. One template can run 200k tokens per call.

All three route through Unity AI Gateway. The chokepoint exists. Until governance is configured, the controls do not.

The issue

Agents are not just LLM calls.

Every hop in an agent loop is a separate cost event and a separate risk surface. The governance problem is wider than token pricing.

LLM calls

The obvious part

  • Input tokens: context, system prompt, tool schemas, conversation history
  • Output tokens: reasoning, code, tool-call arguments
  • Long context loops multiply cost non-linearly
  • Model-tier selection per task is uncontrolled
Tool & API calls

The hidden part

  • Agents call search APIs, GitHub, Jira, internal services
  • Each tool result feeds back into context — more input tokens
  • Unbounded tool loops: retries on error amplify spend
  • No visibility into which tools are called, how often, by whom
Execution actions

The risky part

  • Shell commands, file writes, git push, terraform apply
  • Agents act on infrastructure — not just generate text
  • No approval gate means an agent can act on production autonomously
  • Without a gate, there is no record of who approved it
Design principle: Govern the entire agent surface — not just the LLM endpoint. Cost, capability, and safety controls belong at the same layer.

What it is

Guardrails — what agents can say and do.

Guardrails run on every request and response, before the model sees the input and before the agent sees the output. First match wins.

Input guardrails

Block before LLM

  • Prompts attempting to extract training data
  • System-prompt override attempts (Ignore all previous instructions)
  • Direct requests for internal system paths or service credentials
  • Prompt injection via tool results
Fires before token cost is incurred.
Output guardrails

Redact before agent

  • AWS access-key patterns (AKIA*)
  • Databricks personal access tokens (dapi*)
  • Private IP ranges in generated code
  • Internal hostnames and hardcoded connection strings
Pattern-matched before the agent consumes output.
PII detection

Redact before model

PII in agent context (emails, employee IDs, SSNs, card numbers) is redacted before the LLM processes it.

Redaction is deterministic: the same entity gets the same placeholder across a session.

john.doe@acme.com → [EMAIL_1]
Requirement: All three guardrail types on every agent route. pii_detection: true required to enable built-in PII redaction — guardrails are account-level defaults, not per-team opt-in.
What it is

Approval gates, and policies that remember the session.

Service policies — the approval matrix
Agent actionAutonomous?Requires
read_file✓ Yes—
run_tests✓ Yes—
create_pr✓ Yes—
git push main✗ NoHuman approval
terraform apply✗ NoHuman approval
delete_resource✗ NoHuman approval

Enforced at the gateway, not in agent code

Service policies live in gateway configuration. An agent cannot bypass them — the gateway intercepts the tool call and issues a hold before forwarding.

Contextual policies — four verdicts, not three
ALLOWProceed as requested
DENYBlock the action outright
ASKPause for human approval
TRANSFORMRewrite the action, then proceed

Why four verdicts: policies remember the session

A contextual policy listens to every tool call and model response in a session and accumulates state — it isn’t just evaluating one action in isolation. For cost, that means a running spend total: a soft threshold triggers ASK for confirmation, and a hard cap triggers TRANSFORM — the agent switches to a cheaper model rather than stopping outright. The same session-state mechanism also backs non-cost policies, like scoping an agent’s tools to its stated intent for the session.

Sandbox isolation (Omnigent)

Each coding-agent session gets its own isolated Databricks Sandbox — filesystem and network scoped to that session, credentials injected at session start. A session is shareable by URL for full review.

Rate limits catch loops in real time

A 60 RPM per-engineer coding-agent limit stops a runaway loop within the first minute, well before an unattended loop could run up hundreds of dollars in overnight spend. Engineer sees a 429, investigates, fixes the loop.

Model allowlisting is the cheapest, highest-impact control

An engineer using a premium-tier model for a task a lower tier could handle can cost roughly 15× more for the same output. Allowlisting removes the choice rather than relying on attention.

34% cost reduction from routing alone

Model allowlisting drove a 34% reduction in model spend with no feature changes — the expected range going in was 20–40%.

Why it matters

Rate limits and caps compound

Rate limits stop loops in real time. Caps stop cumulative overruns across the billing period. Both are needed — neither alone is sufficient.

How — Layer 1

Model routing & allowlisting.

Gateway route config
# Serving endpoint config (simplified)
route_config:
  routes:
    - name: coding-agent-senior
      model_allowlist:
        - claude-sonnet-4-*
        - claude-opus-4-*
      tags_filter: {team: engineering, tier: senior}

    - name: completion-tool-mid
      model_allowlist:
        - claude-haiku-3-*
      tags_filter: {team: engineering, tier: mid}

    - name: notebook-agent-data
      model_allowlist:
        - gemini-2-flash-*
      tags_filter: {team: data}
Policy by agent type

Coding agent — senior: Sonnet default, Opus on explicit request. All other model families blocked.

Completion tool — mid-level: Haiku-equivalent tier only. Sonnet, Opus, and Gemini Pro blocked.

Notebook agent — data team: Approved Gemini family, flash by default. All Claude families and GPT-4 blocked.

How — Layers 2 & 3

Rate limits and spend caps — the two responses agents receive.

Rate-limit response (velocity)
# What the agent receives on throttle
HTTP/1.1 429 Too Many Requests
Retry-After: 60
{
  "error": "rate_limit_exceeded",
  "limit": "60 RPM per engineer",
  "reset_at": "2026-07-07T14:32:00Z"
}

First-match-wins: the gateway evaluates most-specific to least-specific. A per-engineer burst limit fires before the team or account limit ever needs to.

Spend-cap response (total)
# Spend cap response — same 429, different cause
HTTP/1.1 429 Too Many Requests
{
  "error": "spend_cap_exceeded",
  "scope": "user",
  "cap_usd": 50.00,
  "spent_usd": 50.12,
  "resets_at": "2026-07-08T00:00:00Z"
}

Finance integration: spend data in system.ai_gateway.usage feeds a nightly roll-up — a line item per team per day, no override path once a cap fires.

How — Layer 5

Service policies — approval gates for high-risk tool calls.

Gateway service policy
# Gateway service policy (simplified)
service_policies:
  - name: high-risk-tool-gate
    match:
      tool_names:
        - git_push_main
        - terraform_apply
        - delete_resource
    action: require_approval
    approval:
      timeout_seconds: 300
      approver_groups: [team-leads]
      on_timeout: deny

What the agent experiences

The gateway intercepts the matching tool call and returns a hold. Execution pauses. A notification goes to the on-call team lead. Approve → agent continues. Deny or timeout → agent receives a rejection. The entire flow is logged.

Enforced at the gateway, not in agent code

Policies live in gateway configuration, not in a prompt or tool definition — an agent cannot reason its way past one.

How — Layer 6

Omnigent sandbox policy — isolation and a per-session spend ceiling.

Sandbox policy YAML
# Omnigent session policy
sandbox_policy:
  isolation:
    filesystem: workspace_only
    network: allowlist
    network_allowlist:
      - api.github.com
      - jira.acme.io
  credentials:
    injection_mode: env_scoped
    rotation: per_session
  spend:
    token_budget_per_session: 200000
    on_budget_exceeded: terminate
  review:
    shareable_session_url: true
    retain_logs_days: 90

Session-scoped, not gateway-scoped

Even when the gateway-level per-engineer cap has headroom, a single session cannot exceed its own sandbox token budget — the session terminates on breach.

Full transparency for high-risk sessions

Any team lead can open a session URL and see exactly what the agent sees: working directory, tool-call history, LLM conversation.

Defense in depth

When an agent tries to write outside its workspace or call an unlisted API, the sandbox rejects it before the gateway is even involved.

How — Observability

The usage table, and what breaks quietly if you skip the fine print.

Queries against system.ai_gateway.usage
-- Daily spend by engineer
SELECT user_id, date_trunc('day', timestamp) AS day,
       sum(total_cost_usd) AS spend_usd,
       sum(total_tokens) AS tokens
FROM system.ai_gateway.usage
WHERE workspace_id = current_workspace_id()
  AND timestamp >= current_date - INTERVAL 30 DAYS
GROUP BY 1, 2
ORDER BY spend_usd DESC;
-- Anomaly alert: 3x daily average breach
WITH daily_avg AS (
  SELECT user_id, avg(daily_spend) AS avg_spend
  FROM (
    SELECT user_id, DATE(timestamp) AS d,
           sum(total_cost_usd) AS daily_spend
    FROM system.ai_gateway.usage
    WHERE timestamp >= current_date - 30
    GROUP BY 1, 2
  ) GROUP BY 1
)
SELECT u.user_id, today.spend, d.avg_spend,
       today.spend / d.avg_spend AS ratio
FROM daily_avg d
JOIN (...today_spend...) today USING (user_id)
WHERE today.spend > d.avg_spend * 3;
Gotchas
IssueDetail
Model-tier creepWithout allowlisting, model choice defaults to whoever pays least attention — a premium model can cost roughly 15× a lower tier for the same output.
Rate limits alone don't cap total spendRate limits bound velocity, not cumulative total. Spend caps are still required.
Session caps are separate from gateway capsA sandbox's per-session token budget can terminate a session even when the gateway-level cap has headroom.
on_timeout: denyAn unanswered approval request denies by default — verify this is the intended failure mode before rollout.
pii_detection: true is not implicitBuilt-in PII redaction requires this flag explicitly set on the route.

The anomaly query can run as a scheduled workflow every few hours — anomalies page the on-call engineer directly. No dashboard required for the urgent case.

How — Rollout sequence

Don't boil the ocean. Start here.

Each step is independently valuable. Stop after any step if the org isn't ready. Later steps get easier once earlier ones are in place.

1
Usage monitoring first

Enable system.ai_gateway.usage and run the spend-by-engineer query. No controls yet — just visibility.

2
Model allowlists

Map teams to model tiers by actual task profile. Expect 20–40% cost reduction within the first billing cycle.

3
Rate limits

Set per-engineer RPM limits conservatively, then tune up. Runaway loops stop; normal use barely notices.

4
Spend caps

Account, team, and daily-user caps. Finance gets a committed ceiling; engineering gets autonomy within it.

5
Guardrails

PII detection and output secret scanning. Security team owns this layer independently.

6
Approval gates & sandbox isolation

Service policies and Omnigent sandboxes for agents with infrastructure access. Pilot with one team, tune, then expand.
