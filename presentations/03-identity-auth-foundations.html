The issue

Every AI app has to answer one question before anything else: who is asking?

Get the token path wrong and you either lose per-user identity — row filters silently return the wrong data — or you introduce a shared credential where none was needed.
3
token paths, one governance model
2
identity worlds: internal & external

Internal teams are the easy half

Employees already authenticate through your workspace IdP. But the harder ask is already on the table: partners, customers, and regulated subsidiaries need the same governed access through their own identity providers — without ever getting a Databricks account.

What this deck answers

Which token path fits which identity world, how authentication and authorization stay separately enforced, and the concrete configuration steps for each path.

What — Native IdP support

Databricks already supports your cloud IdP. Federation is only for identities it doesn't know about.

Azure

Entra ID (your tenant) — default

Users authenticate via Entra. SSO, MFA, Conditional Access all work out of the box.

GCP

Cloud Identity

Google Workspace or Cloud Identity federation. Native SSO.

AWS

Bring your own IdP

Okta, Entra, Ping, or any SAML/OIDC provider via account-level SSO configuration.

Apps ON Databricks → OBO

Users are already authenticated via the workspace IdP. The Databricks Apps proxy injects the user's OBO token automatically. current_user() returns the human email. No extra IdP integration needed.

Apps OUTSIDE Databricks → Federation

An external app — a partner portal, a customer-facing SaaS, an internal tool on separate infrastructure — authenticates users with its own IdP. Those users don't have Databricks accounts. Federation bridges the external IdP via token exchange (RFC 8693).

What — Foundations

Authentication and authorization are two different questions, enforced at two different layers.

Authentication (AuthN) — Who are you?

Proven by OAuth tokens: OBO, M2M, or Federation exchange. The token carries the caller's identity through every layer of the stack.

OBOUser's OAuth token via the Apps proxy
M2MService principal client-credentials grant
FederationIdP JWT exchanged for a Databricks token

Authorization (AuthZ) — What can you do?

Enforced by Unity Catalog at the SQL engine level: row filters, column masks, USE CONNECTION, EXECUTE on functions. Application code cannot bypass these controls.

ScopesLimit what the token can do
UC grantsLimit what the identity can access
BothEnforce independently; either can deny
What — Identity models

Three token patterns. Every resource auth decision reduces to one of these.

OBO (U2M)

User-to-Machine. The platform acts on behalf of a known human.

TokenUser's scoped OAuth via Apps proxy
Identitycurrent_user() = human email
AuditFull per-user attribution

Best for: internal apps, dashboards, agent tools where users have Databricks accounts.

M2M

Machine-to-Machine. A service principal with its own credentials, no human in the loop.

TokenSP OAuth client-credentials grant
Identitycurrent_user() = SP UUID
AuditSP-level only, no human attribution

Best for: background jobs, scheduled pipelines, shared knowledge bases, Lakebase.

Federation

External IdP JWT exchanged for a Databricks token via token exchange (RFC 8693).

TokenIdP JWT → Databricks OAuth token
IdentityRole-based SP, or the user directly
AuditSP-level + app-layer human attribution

Best for: partner portals, CI/CD, external apps that must not hold Databricks secrets.

current_user() = the human's email at every hop. Row filters and column masks fire automatically, with no app code changes. The app never handles login flows, passwords, or OAuth redirects — the platform does all of it.

What — Federation in practice

Federation: an external IdP JWT bridged to a scoped Databricks token.

External UserAuth0 / Okta / Entra tenant
→
Your App Serververifies JWT, maps role
→
/oidc/v1/tokenRFC 8693 exchange
↓

Unity Catalogfires as the role-mapped SP
↓

Your app server is the trust boundary. It verifies the external JWT independently (JWKS signature check) and performs the token exchange server-side — Databricks never trusts the external IdP directly.

What — Service principals

Two identifiers for the same service principal. They are not interchangeable.

Account-level SP

Created at the Databricks account level. Can be assigned to multiple workspaces. Has a globally unique application_id (UUID).

application_idUUID, used in current_user() and UC GRANTs
display_nameHuman-readable label

Workspace-level identity

When assigned to a workspace, the SP gets a workspace-specific principal_id (integer), used for workspace-level ACLs.

principal_idInteger, workspace-specific (e.g. CAN USE on a SQL warehouse)
GroupsAdded to workspace groups for UC governance

Common confusion: application_id is what current_user() returns and what UC GRANTs reference. principal_id is for workspace-level permissions only. In a Federation pattern, each external role maps to a dedicated SP — e.g. sales_west role → fed-exchange-sp-sales-west → sales_west UC group.

Why it matters

The identity model is a governance decision, not just a plumbing detail.

One governance model

Row filters, column masks, and connections enforce identically whether the caller arrived via OBO or Federation. No duplicate engineering effort per identity world.

Audit & compliance

OBO gives full per-user attribution in system.access.audit. That's the difference between an audit log that names a human and one that only names a service principal.

Secretless architecture

Federation and Workload Identity Federation mean no Databricks credentials live in external infrastructure or CI/CD pipelines — fewer secrets to leak, rotate, or forget about.

IdP-agnostic by design: Auth0, Okta, Entra, or any OIDC provider — swap the federation policy config, keep the same architecture and the same audit trail.

Why it matters

Picking the wrong token path fails silently, not loudly.

Wrong path chosen
✕
M2M where OBO was needed. Row filters evaluate against the SP's identity, not the human's — every user sees the same rows. No error, just wrong data.
✕
Databricks credentials in external infra. Every secret checked into a partner's CI/CD config is a credential you now have to rotate, audit, and hope never leaks.
✕
No per-user audit trail. A security review asks “who accessed this row last quarter” and the honest answer is a service principal UUID, not a name.
Right path chosen
✓
OBO for per-user access. Every row filter, column mask, and audit entry names the actual human, automatically.
✓
Federation for external identities. Zero Databricks secrets outside the platform — token exchange is the only bridge, and it's short-lived.
✓
M2M for automation. Background jobs and shared knowledge bases run with fixed, auditable SP permissions — exactly what they should have.

Key insight: all three patterns share the same UC governance, tools, and audit infrastructure. The only difference is how the token reaches your app.

How — Configuring OBO

Databricks Apps handles the OAuth flow. You configure scopes and read one header.

Read the forwarded token
# App code — no login flow, no redirects
token = request.headers[
  "x-forwarded-access-token"
]
workspace_client = WorkspaceClient(
    host=databricks_host,
    token=token,
)
# current_user() now returns the human
Scopes configured in the App UI
sqlSQL warehouse access
genie + dashboards.genieGenie Conversation API (both required)
servingModel Serving endpoints
files.filesFiles & directories
appsCall other Databricks Apps

Configure only what your app needs — least privilege. Scopes are the ceiling: a token without serving cannot call Model Serving even if UC grants would otherwise allow it.

How — Configuring Federation

One token exchange call. Three claims Databricks validates before it trusts the JWT.

Token exchange (RFC 8693)
POST {databricks_host}/oidc/v1/token

grant_type=urn:ietf:params:oauth:
  grant-type:token-exchange
&subject_token={external_idp_jwt}
&subject_token_type=
  urn:ietf:params:oauth:token-type:jwt
&scope=sql genie serving
# scopes are configurable per use case —
# request only what the app needs
What gets validated on every exchange
issMust match the IdP's issuer URL exactly (trailing slash matters)
audMust match the workspace URL registered in the federation policy
subMust match the client ID registered on the target SP

Databricks fetches the IdP's JWKS keys and verifies the RS256 signature independently — it never trusts the caller's own claims. Token TTL: Databricks copies the IdP's exp claim verbatim, typically 1 hour, not refreshable. Plan re-exchange in client code.

How — Gotchas

What breaks quietly if you skip the fine print.

IssueDetail
UI only shows dashboards.genieGenie's API checks for both genie and dashboards.genie. Missing genie returns 403: "required scopes: genie". Patch via the account-level OAuth integration API.
application_id vs principal_idUC GRANTs reference application_id (UUID). Workspace ACLs reference principal_id (integer). Not interchangeable.
M2M mistaken for per-usercurrent_user() under M2M returns the SP UUID, never a human. Don't rely on it for per-user row filters.
Account Federation needs SCIMUsers must be synced via SCIM before account-wide token federation works. Workload Identity Federation (WIF) needs no SCIM — it's SP-only.
iss trailing-slash mismatchA federation policy issuer URL that differs from the IdP's actual issuer by a trailing slash fails validation with a 401, not a helpful error.
