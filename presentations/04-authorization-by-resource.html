Authorization by Resource Type

How access control actually works at each Databricks AI component — serving endpoints, Genie, vector search, functions, connections, tables, and Lakebase.

Companion to Identity & Auth Foundations. That deck covers the token; this one covers what happens once the token arrives at a resource.

Agenda

What this deck covers

1. Why authorization differs by resource, not just by token path

2. Serving endpoints, Genie, and Vector Search — identity model and gotchas each

3. UC Functions and UC HTTP Connections as the safe patterns for writes and external calls

4. Tables — row filters, column masks, and the current_user() vs is_member() trap

5. Lakebase — the one resource that isn't governed by Unity Catalog

6. A decision guide and gotcha checklist for production readiness

The issue

The same token hits a different enforcement model at every resource it touches.

Teams that get the token path right in Identity & Auth Foundations still get burned here — because Genie checks a different scope set than Model Serving, Vector Search has no per-user concept at all, and Lakebase doesn't use Unity Catalog.

What — Serving Endpoints

Agent Bricks and Model Serving: OBO or M2M, but never mix up the API.

Identity model

  • OBO: ModelServingUserCredentials() extracts the user — inside Model Serving only
  • M2M: WorkspaceClient() no-args in Databricks App context
  • Scope required: model-serving
  • Agent Bricks supervisor auto-forwards OBO to up to 20 sub-agents

Two-proxy gotcha

App A calling App B: App B's proxy strips App A's forwarded user token and replaces it with App A's own SP identity.

Fix: use X-Forwarded-Email (proxy-signed, survives hops, cannot be forged) plus M2M SQL with an email-based row filter.

What — Genie Spaces

Genie: OBO is the only supported path, and the scope UI lies to you.

Identity model

  • OBO only — user's token forwarded via X-Forwarded-Access-Token
  • Both genie AND dashboards.genie scopes required
  • current_user() = calling human; is_member() = execution service
  • Rate limit: 5 queries/min (sliding window)

The scope bug

The App integration UI only shows dashboards.genie. The Genie API checks for both scopes.

HTTP 403
"required scopes: genie"

Fix: PATCH /api/2.0/preview/accounts/{account_id}/oidc/custom-app-integration/{id} with {"scopes": ["dashboards.genie","genie","sql"]} — the UI won't show this option.

What — Vector Search

Vector Search: M2M by default, per-user filtering is your application's job.

Identity model

  • M2M only — App SP has SELECT on the index
  • WorkspaceClient() no-args picks up auto-injected SP credentials
  • Scope: vector-search; Delta Sync auto-updates on source change

No native per-user context. The SP retrieves results; your code enforces per-user visibility.

Per-user filtering strategies

Pre-filterInclude a user_group column in the index, pass as a filter in the query. Faster, needs group info indexed.
Post-filterRetrieve top-K, then check each against user permissions. Flexible for complex/external ACLs, wastes retrieval budget.
Shared KBM2M is correct as-is — same docs for every user, no filtering needed.
What — UC Functions

The safe pattern for agent writes: EXECUTE on the function, never raw table grants.

CREATE FUNCTION prod.tools.approve_deal(
  deal_id STRING, approver STRING
)
RETURNS STRING LANGUAGE SQL
RETURN (
  UPDATE prod.sales.deals
  SET status = 'approved',
      approved_by = approver
  WHERE id = deal_id
    AND current_user() IN (
      SELECT email FROM prod.sales.approvers
    )
);

What this achieves

Agent SPHas EXECUTE on the function
Agent SPDoes NOT have UPDATE on the table
FunctionEnforces eligibility via current_user() internally
AuditFunction execution logged to system.access.audit

Functions are the authorization boundary for write operations — the agent never touches the table directly.

What — UC HTTP Connections

Four auth methods, one governed boundary. Only two give identity on both sides.

MethodExternal service seesUse case
Bearer TokenShared (one static token)Simple integrations, static API keys
OAuth M2MShared (app client credentials)Org-level, service-to-service access
OAuth U2M SharedShared (one user's delegated token)Admin-delegated access
OAuth U2M Per UserPer-user identity ✓User-specific (Jira, GitHub as them)
Managed OAuthPer-user identity ✓Google, SharePoint — Databricks handles OAuth

Runtime kill switch. REVOKE USE CONNECTION removes agent access to the tool immediately — no redeploy, no code change.

No read-only scope at the connection level. UC controls who can use the connection, not what they can do through it — create separate connections with different external credentials for read-only vs. read-write.

What — Tables

Row filters and column masks fire everywhere — but only one function reads correctly everywhere.

Row filters & column masks

  • SQL function evaluated at query time, attached to the table
  • Fires regardless of access path: notebook, API, Genie, agent
  • Column masks transform values per user with the same trigger rules

current_user() vs is_member()

Rule: use current_user() everywhere. It's correct in every access path; is_member() breaks silently in Genie and Agent Bricks OBO.

What — Lakebase

The one exception: PostgreSQL-native authorization, not Unity Catalog.

PG-native, not UC

  • AuthN: Databricks OAuth token as PG password, or native PG roles
  • App's SP client ID becomes the PG role name, auto-created on resource binding
  • Role gets CONNECT + CREATE; SELECT/INSERT added manually
NOT Unity Catalog

Two authorization paths, same data

Direct PG connection (apps, psql): PG roles + GRANT + RLS govern access.

Via lakehouse / DBSQL: UC policies apply instead of PG grants.

Same data, different auth model depending on access path.

Why it matters

Getting resource-level authorization wrong doesn't throw an error. It just returns the wrong data.

Risk

A row filter using is_member() in a Genie space leaks data silently — no 403, just the wrong rows returned to the wrong person.

Compliance

Auditors ask which resource enforced which control. “UC row filter” and “PG RLS policy” are different answers for the same-looking table — know which one applies.

Velocity

Wrapping writes in a UC Function once means every future agent that needs the same write inherits the same authorization boundary — no re-derivation per agent.

Authorization is enforced at the resource, not at the application. Every token path from the previous deck still has to clear this layer.

How — Decision guide

Three questions that determine the resource-level pattern.

Question 1

Does the resource use UC for authorization?

Yes — 6 of 7 resources: standard pattern of SP grants, row filters, OAuth scopes.

No — Lakebase only: PG-native roles, GRANT, RLS policies.

Question 2

Does the user need per-user identity at the external service?

Yes — UC Connection with U2M Per User or Managed OAuth.

No — Bearer or M2M OAuth connection is sufficient.

Question 3

Does the agent need to write to a table?

Yes — wrap in a UC Function, grant EXECUTE, not table writes.

No — row filters + column masks on the table directly.
